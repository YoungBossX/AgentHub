## 背景

P6-P14 将 AgentHub 从一个演示循环演进为一个具备目标感知和提供商感知能力的本地编码工作区，包含调度器锁、外部项目工作区、恢复机制、CanonicalSharedContext、提供商指令适配器、提供商分配、AgentProfile 策略、diff/review/preview/staging 部署证据以及安全的自定义智能体草稿元数据。

剩下的产品问题不是另一个 UI 外壳。AgentHub 应允许 Claude Code 或 Codex 在已注册的目标内执行有用的编码工作。目前，规划器和指令构建器仍带有太多演示时代的痕迹：登录页面插槽、狭窄的前端模板，以及对宽泛但合理请求的防御性拒绝。P15 保留了严格的安全边界，但使规划和指令更忠实于用户的请求。

## 目标 / 非目标

**目标：**

- 为有边界的真实编码请求添加 `llm_v1` 规划。
- 在创建任务前，通过 PlanValidator 验证每个 LLM 生成的计划。
- 在透传提供者指令中保留原始用户请求。
- 允许在已注册的目标 `allowedPaths` 内进行有意义的编码。
- 从目标注册表、项目分析器和显式目标配置中推导允许的项目命令。
- 公开计划依据、计划文件、依赖项、目标和验收标准以供审查。
- 验证 Breakout / 打砖块游戏作为最终验收目标，且不添加硬编码的 Breakout 模板。

**非目标：**

- 硬编码的 Breakout 规划器模板。
- 提供商市场。
- 用户创建的任意命令代理。
- 生产环境部署。
- 云部署提供商集成。
- 多用户即时通讯。
- 桌面、IDE 或 CLI 客户端。
- 完整的制品编辑器。
- 替换当前调度器。
- 移除确定性演示兜底。

## 决策

### 决策：LLM 规划器 v1 生成已验证的计划草稿 JSON

`llm_v1` 仅在启用且可用时调用已配置的规划提供程序。其输入应包含原始用户请求、CanonicalSharedContext、目标注册表摘要、项目分析器元数据、最近消息、所选制品引用以及护栏。

规划器的输出必须是结构化的 PlanDraft JSON，包含：

```text
planId
version
plannerMode
userGoal
tasks
roles
targets
dependencies
plannedFiles
rationale
acceptanceCriteria
validationExpectations
guardrailNotes
```

PlanValidator 必须在创建 Task 行之前拒绝无效、不支持、不安全或不完整的 LLM 输出。如果 LLM 规划被禁用或无效，AgentHub 仅在该兜底方案明确适用且已记录时，才可回退到确定性规划。

理由：规划器可以保持灵活，同时不允许未经验证的自由执行。

### 决策：透传指令保留用户意图

对于 `llm_v1` / `passthrough_v1` 计划，提供者指令应足够忠实地保留用户的原始请求和任务描述，以便 Claude Code 或 Codex 理解实际期望的产品。仅当明确选择确定性演示兜底路径时，才应进行演示槽位重写。

提供者指令仍应包含：

- 目标上下文和允许路径；
- 禁止路径与护栏；
- 验证命令；
- 上游 handoff/artifact 引用；
- 验收标准；
- 特定提供商的包装格式。

理由：Claude Code 和 Codex 已经是具备能力的编码代理。P15 应停止将其上下文压缩到旧的演示模板中。

### 决策：权限扩展是目标作用域的

P15 在已注册的目标 `allowedPaths` 内部应更加宽松，而非仓库范围。护栏策略应继续拒绝：

- `.git`；
- `.env` 和 `.env.*`；
- 密钥；
- `node_modules`；
- `.venv`；
- 所选 target/worktree 之外的路径；
- 未经批准的生产部署；
- 未经批准的网络访问；
- 修改 AgentHub 平台代码的普通应用任务。

平台维护保持显式且需审批。

理由：实际编码需要在目标内部编辑组件、样式、配置和测试的空间，但目标注册表仍然是安全边界。

### 决策：项目命令策略来源于目标

允许的验证命令应源自目标注册表、项目分析器和显式目标配置。P15 应支持常见的已配置命令，例如 `pnpm test`、`pnpm build`、`pnpm lint`、`npm test`、`npm run build`、`pytest`，以及为目标声明的特定目标检查命令。

命令输出必须成为诚实的证据。失败的命令不应被改写为成功。

理由：真正的编码验收需要 build/check/test 证据，但命令必须限定目标范围且可审计。

### 决策：Breakout 是验收目标，而非模板

最终请求是：

```text
帮我在当前前端项目里实现一个 Breakout / 打砖块游戏，要求可以用键盘控制挡板，球能反弹，能击碎砖块，有得分、胜利/失败状态和重新开始按钮。
```

规划器可能会将此识别为前端交互式应用请求，但不得使用硬编码的 Breakout regex/template 或脚本化实现。编码提供方应执行具体实现。冒烟测试应验证差异、审查、build/check、预览、预发布部署证据以及浏览器可玩性。

理由：Breakout 足够具体，可以评判端到端质量，但又足够复杂，能够证明 AgentHub 不再局限于登录页面的演示模板。

## 风险 / 权衡

- **风险：LLM 规划器发出不安全或格式错误的任务。**
  缓解措施：在持久化之前，通过 PlanValidator 验证所有输出。
- **风险：透传模式变成任意仓库编辑。**
  缓解措施：将目标注册表的允许路径和拒绝路径作为硬边界。
- **风险：命令在目标策略之外执行。**
  缓解措施：从目标元数据派生命令允许列表，并记录命令证据。
- **风险：Breakout 烟雾被误认为是硬编码模板。**
  缓解措施：明确禁止 Breakout 特定的确定性模板，并要求提供者指令 preservation/evidence.
- **风险：提供者 auth/quota 阻止实际执行。**
  缓解措施：记录精确的标准化提供者错误，不声称成功。

## 迁移计划

1. 添加 LLM Planner v1 接口、配置开关、结构化输出解析以及 PlanValidator 集成。
2. 为 LLM/passthrough 计划添加直通指令模式。
3. 扩展护栏，使其在已注册的目标允许路径内具有宽松权限。
4. 添加项目命令策略和命令证据集成。
5. 在 API/UI 界面中公开规划器理由和任务审查元数据。
6. 运行 Breakout 真实编码冒烟测试并冻结审查。

回滚策略：禁用 `llm_v1`，并将旧的演示请求路由到确定性规划器，同时保持 P15 元数据为只读状态以用于诊断。
