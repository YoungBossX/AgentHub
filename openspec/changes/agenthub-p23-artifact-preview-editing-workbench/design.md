## 背景

P20 改进了制品导航，而 P21/P22 改进了编排和代理配置。AgentHub 现在需要下一层：制品应可作为工作产品被检查和利用，而不仅仅是证据徽章。

当前与制品相关的基本元素包括消息、制品、差异、审查、预览、部署、任务追踪，以及消息编辑器中的选定制品上下文。P23 在这些基本元素的基础上构建，但不会替换调度器、计划验证器、目标注册表或现有的 preview/deploy 证据。

## 目标 / 非目标

**目标：**

- 添加一个制品工作台，能够以一致的元数据和操作渲染常见的制品类型。
- 支持安全、轻量的制品编辑，并创建新的制品版本。
- 支持制品版本历史记录和只读的 compare/restore 元数据。
- 允许用户在后续消息中引用某个制品或章节。
- 保持聊天界面简洁，将详细的制品交互路由到右侧工作台或专注的制品视图。

**非目标：**

- 无完整 Google Docs / Office 编辑器。
- 无二进制 PPT 编辑器。
- 无通过制品编辑直接写入仓库。
- 无生产部署提供商。
- 无任意文件浏览器。
- 无替代 diff/review/preview/deployment 证据。
- 无绕过 Planner、PlanValidator、Scheduler、Target Registry 或 Guardrails。

## 决策

### 制品编辑创建版本，而非仓库写入

编辑制品时应创建新的制品版本记录或带版本的制品载荷。不得直接写入目标仓库。要将编辑后的制品转化为代码变更，用户必须在消息中引用该制品，以便规划器和编码代理通过正常执行路径对其进行处理。

### 渲染器注册表由类型驱动

UI 应将制品元数据映射到渲染器类别：

- `web_preview` 用于健康的 preview/staging URL；
- `markdown_document` 和 `text_document` 用于渲染后的文档类内容；
- `code_snippet` 用于语法感知的代码展示与轻量编辑；
- `diff` 和 `review` 用于已有证据；
- `deployment` 用于 deploy/staging 卡片。

未知的制品类型必须渲染一个安全的 metadata/details 兜底。

### 引用是显式上下文

用户可以在后续消息中附加制品引用或章节引用。消息上下文应包含制品 ID、版本 ID、标题、类型、提供的选定 text/section 以及安全摘要元数据。不得包含受保护的主机路径或机密信息。

### 版本历史保持可审计

每次编辑应记录父版本、编辑器来源、创建时间、摘要以及内容哈希。版本历史会向用户展示，任务追踪证据可引用所选版本。

### 第三方部署仍被推迟

P23 可能展示 deployment/staging 证据并让用户将部署制品交还给代理进行后续处理，但不得新增生产环境的第三方部署提供商。

## 风险 / 权衡

- [风险] 制品编辑可能被误认为代码编辑。 -> 缓解措施：将编辑内容标记为制品版本，并要求代理后续跟进仓库变更。
- [风险] 大型制品可能导致界面卡顿。 -> 缓解措施：使用摘要、截断预览和聚焦详情加载。
- [风险] 渲染器泛滥可能导致用户体验不一致。 -> 缓解措施：使用小型渲染器注册表和安全的兜底渲染器。
- [风险] 引用可能泄露敏感路径或内容。 -> 缓解措施：在上下文元数据中复用现有编辑规则和受保护路径规则。
- [风险] 用户可能期望丰富的 PPT/Word 编辑功能。 -> 缓解措施：P23 仅支持 preview/rendering 和类文本编辑；完整编辑器功能推迟实现。

## 迁移计划

- 以向后兼容的方式添加 version/reference 元数据。
- 现有制品通过兜底或已有的证据卡片继续渲染。
- 如需回滚，隐藏工作台 UI，保留现有的制品卡片、差异对比、预览、审查和部署流程。
