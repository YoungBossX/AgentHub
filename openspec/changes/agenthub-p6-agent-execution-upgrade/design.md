## 背景

AgentHub 有一个冻结的 P4 最终演示基线和一个冻结的 P5 本地单用户 IM 风格多智能体编码工作区 v1 基线。已验证的循环仍然是：

```text
requirement -> plan -> agent execution -> diff -> preview -> mock deploy
```

P5 通过智能体联系人、执行账本、审查制品、执行追踪、有界动态前端规划以及制品消息卡片，改进了工作区界面。剩余的差距在于执行能力：Claude Code 和 Codex 可作为强大的编码适配器使用，但 AgentHub 对于普通请求仍然需要过多显式的 `@orchestrator` 措辞，并且将大多数有用的工作路由到狭窄的规划器模板中，这些模板将任务简化为原始的演示应用路径。

P6 侧重于在添加更多 UI 封装之前，将 AgentHub 打造成一个实用的代理执行路由器和协调器。预期的升级是渐进式的：首先是消息路由、直接分配和编排器自动运行，其次是上下文和指令质量，然后才是一个有边界的全栈垂直切片。

## 目标 / 非目标

**目标：**

- 默认将未明确提及角色的消息路由至编排器/管理器。
- 让 `@frontend`、`@backend`、`@qa` 和 `@review` 作为显式高级分配快捷方式，直接创建可执行任务。
- 通过现有 TaskRun 路径自动启动编排器创建的安全演示目标编码任务。
- 在直接分配任务中保留用户的原始请求。
- 构建会话上下文包，为适配器提供当前工作区状态。
- 为前端、后端、QA/review 和管理器任务生成角色特定指令。
- 添加或规划安全演示后端目标，使后端执行默认不修改 AgentHub 平台后端。
- 为有界全栈应用生成添加契约优先编排。
- 使用真实或诚实标注的兜底证据验证一个迷你应用垂直切片。
- 保留 P4/P5 差异、预览、审查、制品卡片和模拟部署基线。

**非目标：**

- 任意 SaaS 生成。
- 对 AgentHub 平台代码库的无限制编辑。
- 生产部署或真实部署提供商。
- 多用户即时通讯或外部即时通讯集成。
- 提供商市场、Docker 沙箱、PR 创建或企业审批。
- 支付、认证、多租户或合规性要求较高的生产系统。
- 在冲突处理机制存在之前，同一会话内的并行写入执行。

## 决策

### 决策 1：将编排器设为默认路由

未明确指定角色的消息默认将路由至编排器/管理器。编排器会决定是以对话方式回答、创建frontend/backend/QA/review任务、生成contract/task图表、提出澄清性问题，还是如实拒绝不受支持的请求。

选中的制品上下文遵循同样的规则：它默认作为上下文传递给编排器，而不是绕过编排器并隐式地将工作分配给某个角色。

考虑的替代方案：要求用户始终输入 `@orchestrator`。这虽然保持了路由的明确性，但会让正常的产品使用体验显得不自然，并强化了当前演示专用的提示风格。

### 决策 2：将角色提及视为显式分配的快捷方式

显式提及具有最高优先级。`@orchestrator` 路由到管理器，
而 `@frontend`、`@backend`、`@qa` 和 `@review` 在请求可限定到受支持目标时，
直接创建角色分配的任务。这些提及是面向已知应由哪个代理负责工作的用户的
高级分配模式。

考虑的替代方案：即使用户明确提到了角色，也通过编排器发送每条消息。这虽然能简化路由，但会使角色提及基本成为装饰性功能，并降低高级用户的工作流速度。

### 决策 3：自动启动编排器创建的安全演示任务

当编排器创建安全的演示目标编码任务时，AgentHub 将通过现有的 TaskRun 生命周期自动启动该任务。这是一个自主性增强功能，而非完整的 approval/risk 系统。自动启动边界保持狭窄：在此切片中仅限安全的演示前端任务，后端自动启动将推迟至演示后端目标存在时。

考虑的替代方案：始终要求用户点击“开始”。这保留了手动控制，但使得评估编排器是否能够真正将代理执行作为产品体验来驱动变得困难。

### 决策 4：在直接任务中保留原始用户请求

直接任务会将原始 message/request 与派生角色、目标和安全元数据一同存储。指令构建器可以添加约束和上下文，但不得将所有请求都简化为旧的仅限登录页面的措辞。

考虑的替代方案：继续将请求转换为固定模板。这虽然能保持演示稳定，但会阻碍有意义的智能体执行能力测试。

### 决策 5：引入会话上下文包边界

适配器将接收一个结构化的上下文包，其中包含最近的消息、
执行记录、选定的制品、最新的差异元数据、变更的文件、
preview/deploy 状态、当前目标以及相关的合约数据。该上下文
包成为聊天状态与适配器提示之间的边界。

已考虑的替代方案：让每个适配器临时获取上下文。这可能导致提示不一致、跨会话泄漏以及难以测试的行为。

### 决策 6：使用基于角色的指令构建器

前端、后端、QA/review 和管理员角色需要不同的指令：

- 前端指令聚焦于 `apps/demo` UI 文件和可预览性；
- 后端指令聚焦于演示后端目标，而非 AgentHub 自身的 FastAPI 应用；
- QA/review 指令聚焦于差异、契约、风险以及非阻塞性发现；
- 管理者指令聚焦于契约、任务图、依赖关系，以及在遇到不支持情况时的澄清。

已考虑的替代方案：向每个适配器传递通用指令。这会使角色分配基本流于形式，且不会改善 backend/review 的质量。

### 决策 7：添加一个安全的演示后端目标

P6 后端执行应针对演示后端，例如 `apps/demo-api`。
后端 Agent 任务不得随意修改 `apps/api`，因为它是
AgentHub 平台后端，属于控制平面的一部分。

考虑的替代方案：允许在 `apps/api` 内部执行后端任务。这虽然能提升速度，但会模糊被测产品与平台代码的界限，从而增加控制平面的风险。

### 决策 8：采用契约优先的全栈编排

对于有边界的小型应用生成，编排器会首先创建一份结构化契约，供后端和前端任务引用。这可以通过显式的 `@orchestrator` 提及触发，也可以通过默认路由到编排器的普通未提及请求触发。该契约定义了应用目标、实体、API 端点、UI 界面、验收标准以及目标目录。

考虑的替代方案：让前端和后端任务独立解释相同的用户提示。这种方式速度快，但会导致 API 行为与 UI 预期之间出现偏差。

### 决策 9：保持同会话写入任务串行

P6 可能会将前端和后端任务规划为独立步骤，但在冲突检测、锁机制、合并策略和恢复用户体验就绪之前，同一会话中的写入任务仍保持串行执行。

考虑的替代方案：允许立即并行执行。这虽然能带来更令人印象深刻的演示，但存在破坏共享工作树并产生冲突差异的风险。

## 风险 / 权衡

- **风险：直接提及执行会引发宽泛且不安全的请求。** 缓解措施：将每个直接任务绑定到受支持的角色、安全的目标目录、受保护的路径以及诚实的 unsupported/clarification 状态。
- **风险：自动运行会启动不安全的工作。** 缓解措施：将 P6-1 自动运行限制为编排器创建的、安全的演示目标编码任务，并保留受保护的路径、命令白名单和适配器护栏。
- **风险：默认编排器路由会隐藏用户意图。** 缓解措施：显式角色提及具有最高优先级，并且所选制品上下文会包含在编排器上下文包中。
- **风险：限制较少的指令会降低演示的确定性。** 缓解措施：保持 P4/P5 路径完整，保留兜底执行，并为指令构建和路由添加针对性测试。
- **风险：后端目标脚手架会增加项目复杂性。** 缓解措施：有意将 `apps/demo-api` 保持小型、本地化，并与 AgentHub 的平台 API 分离。
- **风险：契约优先的规划会变得过度构建。** 缓解措施：首先支持一个有限的小型应用系列，使用小型模式并明确非目标。
- **风险：审查输出显得具有权威性。** 缓解措施：除非后续变更实现了门控，否则保持 P6 审查为建议性质。
- **风险：上下文包会在会话之间泄露数据。** 缓解措施：仅从当前会话记录构建包，并测试会话隔离。

## 迁移计划

1. 添加消息路由策略：优先显式提及，否则默认路由到编排器/管理器。
2. 通过现有 TaskRun 路径，为安全的演示目标任务添加窄范围编排器自动运行功能。
3. 添加上下文包生成功能，并在指令中使用该功能，同时保留现有 TaskRun 生命周期。
4. 添加基于角色的指令构建器及每个角色的测试。
5. 添加演示后端目标并更新安全目标元数据。
6. 为单个有界全栈迷你应用添加契约优先的编排器输出。
7. 演练完整的 P6 垂直切片，仅在验证通过后冻结。

回滚操作很直接，而 P6 是任务范围内的：禁用直接提及任务创建，并回退到现有的 P5 planner/task 路径。
