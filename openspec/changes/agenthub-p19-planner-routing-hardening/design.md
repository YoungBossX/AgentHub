## 背景

P17 引入了以 LLM 为先的对话路由，P17b 增加了基于 API 的计划器提供程序。随后 P18 使记忆和指令变得可审计。P18c 暴露了计划器层仍然不一致的问题：

- API 规划器提供者使用较弱的对话路由提示。
- Claude CLI 规划器使用更严格的提示，指示编程请求返回 `task_plan`，并包含预期的 `planDraft` 合约。
- `planning.py` 将任何有效的非 `task_plan` LLM 结果视为最终结果并返回 `[]`，因此确定性兜底仅在 LLM 被禁用、不可用或抛出异常时运行。
- 误路由大多不可见。用户看到聊天消息，但 Task、TaskRun、任务追踪和规划器证据并未明确说明编码请求被 `assistant_reply` 吞没。

P18c 图书馆管理应用请求证明，长期记忆可以被检索，但有效的 `assistant_reply` 结果仍可能阻止任务创建。这使得 AgentHub 即使在存在真正的规划器和编码代理基础设施后，仍感觉受模板束缚。

## 目标 / 非目标

**目标：**

- 让 API 规划器提供者和 Claude CLI 规划器使用同一个规范化的规划器合约和提示词来源。
- 对于安全且目标范围明确的构建/实现/创建/开发/修改软件请求，明确要求使用 `task_plan`。
- 对于普通聊天、问候和能力询问，保留 `assistant_reply`。
- 对于模糊或不安全的请求，保留 clarification/refusal/approval 行为。
- 让确定性规划器作为兜底和回归基线，而非主要能力边界。
- 记录规划器路由错误和验证失败作为证据。
- 支持新颖的应用请求（如图书馆管理应用），前提是这些请求能基于已注册或可准备的目标。

**非目标：**

- 不要硬编码图书馆管理应用模板。
- 不要在 P19 中移除确定性兜底。
- 不要移除目标注册表、PlanValidator、Scheduler、Guardrails 或运行时配置。
- 不允许任意的主机路径写入。
- 不要让 API 规划器提供者执行代码。
- 不要实现提供者市场、新的编码适配器、生产部署或云令牌管理。

## 决策

### 使用一个规范的规划器提示词源

在 `planner_contracts.py` 中或附近创建一个共享的规划器提示构建器，供 API 规划器提供者和 Claude CLI 规划器共同使用。规范提示必须包含：

- Planner LLM 作为意图路由器和 PlanDraft 生成器的角色，而非编码代理；
- 结果定义；
- 编程请求必须返回 `task_plan` 的规则；
- 完整的 `ConversationOutcome` 和 `PlanDraft` 字段预期；
- 允许的角色、意图类型、制品类型、风险等级和审批标志；
- 明确说明普通聊天不会创建任务；
- 少量示例：
  - 问候 → `assistant_reply`；
  - 图书馆管理应用/库存类请求 → `task_plan`。

考虑的替代方案：仅修补 API 提供商的提示词。这样做工作量较小，但会导致 Claude CLI 和 API 的提示词成为两个逐渐偏离的真实来源。

### 将固定模块视为兜底基线

`login page`、`todo`、`notes`、`mini-crm` 和 `Breakout` 仍然是有效的回归测试示例。它们不应定义 AgentHub 的能力上限。

新应用请求应在以下情况下可执行：

- 意图为编码任务；
- 目标已注册或可通过已批准的工作区流程准备；
- 计划路径在允许路径范围内；
- 命令符合项目命令策略；
- 风险与审批要求通过 PlanValidator 验证。

考虑的替代方案：立即删除确定性模块。这将使 LLM 可用性成为唯一路径，并削弱 demo/recovery 的稳定性。

### 将硬性非任务短路替换为 LLM 优先的兜底

当 LLM 返回 `assistant_reply` 时，路由应区分：

- 纯聊天 → 创建编排器回复并停止；
- 安全可执行编码请求 → 继续进入确定性兜底或更严格的修复提示；
- unsupported/unsafe 请求 → 视情况执行 reply/refuse/approval。

P19 应优先使用确定性兜底，因为它已使用目标注册表和现有的安全路径检查。如果范围限定且有边界，可以添加修复循环，但绝不能从未经验证的输出创建任务。

考虑的替代方案：始终重试 LLM。这可能会提高质量，但会增加延迟和提供商成本，并且仍然需要确定性的安全检查。

### 使误路由可观测

规划器证据应记录：

- 可用的 LLM 提供商 id/type/source/model/preset；
- LLM 返回的结果类型；
- 确定性兜底是否认为请求可执行；
- 兜底原因，例如 `non_task_coding_outcome`；
- 模式验证错误和 PlanValidator 错误，包括安全摘要；
- 兜底创建任务时生成的任务 ID；
- 不包含密钥、原始 API 密钥或受保护路径。

此证据应在任务计划 JSON、任务追踪以及日志或事件中可见，前提是现有模式允许。

考虑的替代方案：仅在服务端记录日志。这有助于运维人员，但无法使会话证据具备可审计性。

### 将目标注册表作为执行边界

P19 不应为了处理新颖的应用路由而放宽目标安全性。新项目在写入执行之前，仍必须成为外部目标。如果某个请求想要创建一个新的桌面项目，而目标尚未 registered/preparable，路由应返回澄清说明或准备目标指令，而非写入任意路径。

对于 P18c，桌面排练目标已准备就绪，因此图书馆管理应用请求应在提示或兜底加固后成为外部前端任务。

## 风险 / 权衡

- [风险] `assistant_reply` 后的兜底可能为普通聊天创建任务。→
  缓解措施：要求非聊天的可执行信号和活跃的安全目标检查。
- [风险] 兜底可能隐藏提供者的不当行为。→ 缓解措施：始终记录
  `plannerSource: fallback`、`reason: non_task_coding_outcome` 和原始
  LLM 结果。
- [风险] 强提示仍可能被弱模型忽略。→ 缓解措施：
  模式验证、确定性兜底、可选修复循环和指标。
- [风险] 新型应用支持可能将写入范围扩展得过宽。→ 缓解措施：
  将目标注册表、允许路径、拒绝路径、命令策略和
  PlanValidator 作为硬性关卡。
- [风险] 测试变得特定于提供者。→ 缓解措施：使用虚假规划器客户端
  进行契约测试，并将真实提供者的冒烟测试保留给冻结审查。

## 迁移计划

1. 添加共享规划器提示构建器，并更新各提供者实现以使用该构建器。
2. 添加测试，证明 API 和 Claude CLI 提供者包含相同的任务规划合约语言。
3. 调整 `planning.py` 非任务结果处理，以允许对安全可执行请求进行确定性兜底。
4. 为非任务误路由和验证失败添加规划器证据。
5. 通过强化后的规划器重新运行 P18c 图书馆管理应用路由。
6. 如果真实编码提供者仍不可用，则将其记录为提供者运行时证据，而非规划器路由失败。

回滚很简单：撤销 P19 的代码变更，恢复到现有的以 LLM 为先的行为，仅在提供商故障时执行确定性兜底。

## 待定问题

- P19 是否应包含有边界的 LLM 修复循环，还是在确定性兜底证据稳定后将修复循环留给 P20？
- P19 中的新项目目标准备是否应自动进行，还是 P19 仅路由到已准备好的外部目标，并将自动注册留给后续变更？
