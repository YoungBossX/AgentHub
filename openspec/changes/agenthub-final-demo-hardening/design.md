## 背景

最新重新评估和 P4-0 审计表明，AgentHub 最准确的描述是
本地单用户 Agent 编码工作区 / 强演示 MVP。已验证的核心循环是：

```text
requirement -> plan -> agent execution -> diff -> preview -> mock deploy
```

存在以下证据：

- 通过 `claude_code` 实现真实的本地代理执行；
- 强制 Codex 失败并配合 `ScriptedMockAdapter` 兜底路径；
- 同一会话内的后续文本变更路径；
- 持久化的差异、预览和模拟部署制品。

剩余风险并非演示缺少产品界面。风险在于
基线漂移：旧的 P0 护栏说一套，当前代码和文档说另一套，
最终的演示仍需一次浏览器点击排练、
可重复的重置方案，以及对 AgentHub 是什么、不是什么给出清晰解释。

## 目标

- 使仓库的治理文档与当前实现保持一致，同时不隐藏原始的 P0 兜底路径。
- 保持最终项目身份的真实性：本地单用户 Agent 编码工作区，而非完整的 Feishu/WeChat-style 协作平台。
- 尽可能通过浏览器 UI 验证最终演示，而不仅限于后端 API 调用。
- 确保演示重置与恢复安全、明确且可重复。
- 捕获一份简洁的最终检查清单，包含真实 Agent、兜底、后续跟进及验证证据。
- 准备一份最终架构与面试说明，明确列出各项限制。

## 非目标

- 除非后续任务明确发现文档、重置工作流或验证无法解决的阻碍，否则不要在此加固方案中修改应用代码。
- 不要实现完整的即时消息协作、多用户在线状态、外部聊天集成、生产环境部署、提供商市场、Docker 沙箱、WebSocket/multiplayer 传输、PR 创建、广泛的任意编辑或企业审批流程。
- 除非实际执行了真实的浏览器点击演练，否则不要声称浏览器自动化成功。
- 当部署卡片由模拟后端支持时，不要声称真实部署成功。

## 设计方法

### 1. 基线治理清理

将当前实现现状视为基线：

- P0 兜底路径保持保留且演示安全。
- P1/P2/P4 新增内容（如 `ClaudeCodeAdapter` 和 `AGENTHUB_DEFAULT_CODE_ADAPTER=claude_code`）是当前现实，而非未来推测。
- 该项目应描述为具备有力证据的本地单用户演示 MVP，而非完整的即时通讯协作产品。

清理工作应更新过时的护栏措辞，但不得放宽重要的安全边界：

- 不保留 WebSocket、不保留 Docker 沙箱、不保留提供商市场、不保留生产环境部署、不保留外部即时通讯集成、不保留广泛的平台扩展；
- 明确保留基于兜底的 P0 路径；
- 明确保留 command/path/network 护栏。

已知的严格 OpenSpec 问题应通过将旧变更中无效的 `SHOULD` 需求措辞改为严格兼容的形式，或以其他方式重构该需求以使 `openspec validate` 能够通过来解决。

### 2. 浏览器端到端点击排练

演练应优先使用真实的浏览器 UI 路径：

1. 使用文档中的命令启动 API 和 Web 应用。
2. 在浏览器中打开 AgentHub。
3. 创建或选择一个干净的会话。
4. 发送 `@orchestrator build a login page for the demo app`。
5. 从 UI 启动一个真实的 Agent 运行。
6. 验证可见的任务状态变化。
7. 从 UI 检查差异卡片。
8. 从 UI 启动预览，并确认 iframe 显示更新后的 Vite 应用。
9. 从 UI 创建模拟部署卡片。
10. 重新加载并确认持久化的消息、任务、运行和制品仍然可见。

如果自动化浏览器控制被本地权限阻止，任务必须明确说明这一点，并准确记录哪些部分是通过人工验证、通过面向 UI 的端点验证，或未经验证。

### 3. 演示重置 / 清理种子辅助工具

重置工作流默认必须是非破坏性的：

- 重置前备份 `apps/api/data/agenthub.sqlite3`；
- 记录足够的元数据以恢复先前状态；
- 运行现有的数据库 initialization/seed 路径；
- 如果无需修改应用代码即可完成，则创建或记录一个干净的演示会话种子；
- 除非实现任务明确证明清理是安全的，否则避免删除已注册的 Git 工作区；
- 记录恢复说明。

如果后续添加辅助脚本，应遵循现有项目约定，且不得静默安装依赖项。

### 4. 最终检查清单与总结

清单应以证据为先：

- 真实代理路径；
- 兜底路径；
- 跟进路径；
- browser/UI 验证级别；
- SSE/reload 恢复级别；
- 差异、预览和模拟部署证据；
- 验证命令；
- 剩余注意事项。

最终项目总结应说明：

- 前端、后端、持久化与执行架构；
- 适配器模型及兜底机制存在的原因；
- 从 TaskRun 事件到 diff/preview/deploy 卡片的制品流水线；
- 基于会话工作树的隔离模型；
- 有意推迟处理的内容。

### 5. 最终冻结审查

冻结审查应验证：

- 加固任务已完成并勾选；
- 最终文档相互一致；
- 基于兜底的 P0 演示保持完好；
- 验证命令已执行并记录；
- 未意外引入应用代码功能扩展；
- 任何剩余的脏文件均已说明；
- 基线已准备好，若用户后续要求可进行标记或归档。

## 风险与缓解措施

- **浏览器自动化可能在本地被拦截。** 通过记录诚实的浏览器手动排练记录和面向 UI 的 API 证据来缓解，而不声称自动点击。
- **真实的代理执行依赖于本地 CLI 认证、配额和可用性。** 通过保留 ScriptedMockAdapter 兜底路径并记录真实路径是否被使用来缓解。
- **重置辅助工具可能破坏有用的本地证据。** 通过备份、恢复说明以及默认不进行破坏性工作区清理来缓解。
- **文档漂移可能持续存在。** 通过让 P4-1 一次性更新所有基线文档，并将最终冻结审查作为一致性检查来缓解。
