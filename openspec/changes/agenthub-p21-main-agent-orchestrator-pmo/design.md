## 背景

P15-P19 强化了规划器和路由路径，P20 让工作区更易用，调度器已具备依赖、目标锁定、过期运行、重试和兜底状态。目前仍缺少一个连贯的主代理/PMO 层，用于将这些内部状态转化为计划决策和下一步行动。

P21 不应将编排器变成第二个调度器。它应位于 Planner、PlanValidator、Scheduler、Guardrails、Target Registry 和 TaskRun 执行之上，作为协调与解释层。其职责是帮助用户理解计划、在需要时批准或拒绝执行、检查阻塞项，并选择安全的恢复操作。

## 目标 / 非目标

**目标：**

- 引入明确的 PMO 计划决策状态与证据。
- 新增计划评审决策 API，支持批准、拒绝和澄清操作。
- 展示可并行执行的任务及 dependency/blocker 摘要。
- 在任务追踪中展示重试、兜底、审批和冲突等下一步操作。
- 通过 PlanValidator 和 Scheduler 确保所有可执行工作均经过验证。
- 在 UI 中使目标锁定和脏工作区冲突更易于理解。

**非目标：**

- 不进行自动代码冲突合并。
- 未经验证不得随意编辑计划。
- 不得替换调度器、计划验证器、目标注册表、防护栏或提供者分配。
- 不新增编码适配器。
- 不采用多用户项目经理审批工作流。
- 不进行生产环境部署或外部即时通讯集成。

## 决策

### 将 PMO 决策状态首先存储在任务计划元数据中

P21 可以在每个任务的 `plan_json` 中，通过结构化的 `pmoDecision` 或 `planDecision` 块来表示计划决策状态。这避免了模式迁移，同时通过现有的任务响应和任务追踪保留了可审计性。

如果后续阶段需要协作式计划编辑或决策历史记录，可以引入专用表。

考虑的替代方案：立即创建新的 `PlanReview` 表。从长期来看这样更清晰，但在所需行为得到验证之前，会增加迁移和 UI 范围。

### 添加窄决策端点，而非直接修改原始计划 JSON

暴露明确的审批、拒绝和请求澄清端点。服务器应仅更新已批准的决策字段和调度器状态。在 P21 阶段不得接受对计划 JSON 的任意变更。

已考虑的替代方案：一个通用的 PATCH `/tasks/{id}/plan`。这过于宽泛，且存在绕过验证的风险。

### 从现有 scheduler/task/run 状态推导下一步操作

下一步行动应基于现有事实计算：任务状态、调度器元数据、任务运行记录、审批请求、兜底可用性以及任务追踪阻塞项。PMO 的下一步行动属于指导性建议，不具备执行权限。实际的重试、兜底重试、审批及运行启动仍通过现有端点执行。

考虑的替代方案：将下一步操作持久化为独立记录。这可能导致指导信息过时，除非每次状态转换都更新它们。

### 保留冲突处理的检查与选择

当目标锁定或工作区脏冲突出现时，PMO 应解释被阻塞的内容，并提供安全操作，如检查、刷新、依赖项完成后重试，或要求用户自行解决。它不得合并或丢弃用户的更改。

考虑的替代方案：自动暂存或自动合并冲突。这与现有的安全策略相冲突，可能导致用户工作丢失。

### 将并行性呈现为就绪组

UI 应将任务分组为：就绪、等待依赖、等待目标锁定、阻塞、运行中和已完成。在 P21 阶段不应引入拖拽式 DAG 构建器。

考虑的替代方案：构建一个完整的图形编辑器。这个范围过于宽泛，更适合留到未来需要时，在可视化工作流阶段再实现。

## 风险 / 权衡

- [风险] 仅包含元数据的决策状态可能难以查询。 -> 缓解措施：
  保持结构稳定并通过响应暴露；仅在必要时后续迁移至表格。
- [风险] 用户看到计划审查后可能期望进行计划编辑。 -> 缓解措施：
  优先支持 approve/reject/clarification，并将任意编辑标记为后续验证阶段。
- [风险] 下一步操作可能与实际调度器状态不一致。 -> 缓解措施：
  根据当前 scheduler/task/run 状态计算而非持久化存储。
- [风险] 冲突解释可能不完整。 -> 缓解措施：包含目标
  ID、原因、锁持有者 ID、脏文件摘要及安全的下一步操作。
- [风险] PMO 层可能意外自动启动不安全的工作。 -> 缓解措施：
  所有执行仍受 Scheduler、PlanValidator、目标策略及现有 TaskRun 端点管控。

## 迁移计划

1. 添加 PMO 决策元数据辅助函数及测试。
2. 为任务计划决策添加窄化 approve/reject/clarification 端点。
3. 在任务追踪或任务响应中扩展 PMO 后续操作，这些操作源自调度器状态。
4. 更新任务时间线和任务面板 UI，以显示计划决策、并行就绪状态、阻塞项及后续操作。
5. 为每个行为添加聚焦的 backend/frontend 测试。
6. 冻结并完成完整的 P6-P20 基线验证。

回滚很简单：移除 PMO 决策端点和 UI，同时保留现有任务、任务运行、调度器和任务追踪行为不变。

## 待定问题

- 澄清请求是立即创建编排器消息，还是仅将任务标记为需要澄清，等待用户发送后续消息？
- P21 是允许一次性批准整个多任务计划，还是要求对首个版本逐任务审批？
- PMO 的下一步行动是仅从任务追踪中返回，还是也应嵌入到每个任务的响应中？
