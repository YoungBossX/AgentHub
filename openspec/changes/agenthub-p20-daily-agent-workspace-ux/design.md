## 背景

AgentHub 的后端已发展为一个功能完备的本地单用户 Agent 编码工作区：LLM 路由、目标注册表、运行时配置、内存快照、提供商分配、调度器状态、证据制品、预览以及本地暂存部署均已就绪。前端展示了这些组件，但主界面仍像演示控制面板：用户必须将会话、任务运行、制品、运行时配置和任务追踪理解为各自独立的内部系统。

P20 将该界面转变为日常使用的工作台。它应使日常使用变得清晰：选择会话、与编排器对话、了解对话是直接式还是群组式、查看哪些目标和智能体处于活跃状态、附加上下文、检查 plan/evidence 并继续任务。在现有核心变得更易用之前，不应创建庞大的新平台层。

## 目标 / 非目标

**目标：**

- 让聊天页面成为主要工作区，而非设置面板。
- 为用户提供更清晰的会话列表，包含 search/filter 和状态摘要。
- 在不改变后端路由语义的前提下，使直接/群组对话模式可见。
- 让选中的 message/artifact 上下文在发送前可见且可移除。
- 以可读的、非编辑优先版本展示任务计划审查元数据。
- 在主工作区中展示 agent/provider/runtime/memory/evidence 状态。
- 改善从任务和右侧面板发现和导航制品的能力。
- 保留现有的 P6-P19 执行、目标、内存、运行时和证据行为。

**非目标：**

- 不支持多用户聊天或实时协作。
- 不提供完整的计划编辑器或计划变更 API；P20 阶段计划审查仍为只读。
- 不提供 web/docs/PPT/code 的完整制品编辑器；P20 仅准备更清晰的导航和上下文交接。
- 不新增编码适配器、OpenCode 集成、提供商市场、云部署或生产部署。
- 不允许用户随意创建 shell 命令代理。
- 不替换调度器、计划验证器、目标注册表、运行时配置或内存快照。

## 决策

### 保持主页为三区工作台

现有布局已包含左侧 session/sidebar、中央 chat/task 时间线以及右侧制品面板。P20 应优化此结构而非重写应用外壳。左侧区域变为会话导航与顶层设置入口，中央区域保持对话与任务计划，右侧区域变为 evidence/artifact 工作台。

备选方案：为会话和制品分别引入独立路由。这会在核心日常工作流程顺畅之前增加导航复杂度。

### 优先将 Direct / Group 视为 UI 模式

AgentHub 已支持提及和角色分配。P20 应展示“直接”和“群组”模式作为面向用户的对话框架，但不要发明新的聊天后端模型。直接模式聚焦于编排器或特定角色提及；群组模式则强调编排器加上活跃的角色代理和任务图。

考虑的替代方案：立即添加新的 `conversation_mode` 数据库字段和路由分支。如果用户需要超出 UI 框架的持久模式语义，可以等到 P21 再处理。

### 在计划编辑前添加只读计划审查

后端已暴露 `planReviewMetadata` 和任务计划 JSON。P20 应使这些元数据在任务时间线中可读：理由、角色、目标、依赖项、计划文件、验证期望和验收标准。编辑或批准修改后的计划需要后续的 API 约定，属于 P21 范畴。

已考虑的替代方案：现在构建一个完整的计划编辑器。除非先精心设计后端 API，否则这可能会绕过或重复 PlanValidator 的功能。

### 使用现有消息和制品获取上下文

P20 应改进合成器和消息操作中的选定上下文显示，而无需新增上下文数据库表。引用的消息和选定的制品可继续通过消息内容和现有制品标识符流转，直到 P23 引入更丰富的制品引用和章节级选择。

考虑的替代方案：现在引入正式上下文篮子模型。该方案虽然有用，但最好在 P23 阶段设计好制品 selection/version 语义后再实施。

### 仅在 UI 无法推导状态时，才优先考虑小型 API 扩展

大多数状态数据已存在于会话、任务、任务运行、账本、任务追踪和运行时元数据中。P20 应首先从现有响应中推导出 UI 摘要。如果缺少某个小字段，应将其添加到现有响应中，而不是引入宽泛的新端点。

考虑的替代方案：创建一个专门的工作区仪表盘端点。
这在将来可能会很有用，但在 UI 状态结构稳定之前还为时过早。

## 风险 / 权衡

- [风险] 仅限 UI 的直接/群组模式可能显得肤浅。-> 缓解措施：将其标记为对话框架，并在需要时将持久路由语义保留到 P21。
- [风险] 更多状态徽章可能使聊天页面变得杂乱。-> 缓解措施：保持摘要紧凑，并将详细配置移至设置路由。
- [风险] 只读计划审查可能让希望编辑计划的用户感到沮丧。-> 缓解措施：明确只读边界，并为 P21 提供可编辑的审批计划。
- [风险] 制品导航改进可能看起来像编辑支持。-> 缓解措施：将 P20 标签保持在 inspection/context 交接附近，而非编辑功能。
- [风险] 小型 API 变更可能漂移到产品范围。-> 缓解措施：仅添加工作区显示直接需要的字段，并保留模式。

## 迁移计划

1. 在广泛样式化之前，为会话过滤、模式显示、上下文显示、计划审查元数据和制品导航添加聚焦的组件测试。
2. 就地优化现有工作区组件。
3. 仅当经过测试的 UI 路径无法使用现有数据时，才添加最小化的 API/schema 暴露。
4. 每完成一个任务后，更新文档和 OpenSpec 任务状态。
5. 为每个任务运行针对性测试，然后在冻结时运行完整的 `pnpm check`、`pnpm test`、`pnpm demo:api:test`、`git diff --check` 以及严格的 OpenSpec 验证。

回滚很简单：回退 P20 UI 提交以恢复当前工作区外壳，同时保持后端执行行为不变。

## 待定问题

- 在 P21 阶段，持久化的 Direct / Group 模式应存储在每个会话中，还是在多用户协作功能出现之前保持为本地 UI 偏好？
- 在 P21 阶段，计划编辑应直接操作 Task 记录，还是创建一个独立的提议计划制品，在任务变更前需要审批？
- 在 P23 阶段，制品工作台应对所有文本制品使用 Monaco，还是使用特定类型的 renderer/editor 模型？
