## 背景

AgentHub 目前拥有一个强大的本地编码工作区基线：

- P6 通过编排器、后端代理、前端代理、差异对比、审查、预览和模拟部署，验证了有界全栈小型 CRM 的执行能力。
- P7 通过目标注册表，使规划、指令、审查和平台模式具备目标感知能力。
- P8 增加了依赖感知调度和目标锁机制。
- P9 增加了外部本地项目工作区模式。
- P10 增加了心跳检测、过期锁清理、检查点、重试幂等性、故障传播、冲突检测以及可审计的恢复操作。

部署仍仅限模拟。模拟部署对演示很有用，但它无法证明生成的前端输出能够被构建、提供并作为真实的预发布制品打开。P11 添加了一个真实的本地预发布部署提供程序，同时将生产部署保留为明确的非目标。

关键约束：

- AgentHub 仍为本地单用户 Agent 编码工作区。
- 目标注册表仍是目标路径策略和运行时命令的真实来源。
- 预发布部署必须明确标注为预发布，绝不能是生产环境。
- 现有模拟部署行为必须在需要时保持可用。
- 失败的部署和被阻止的部署关卡必须如实记录。
- P11 不得引入云令牌、外部提供商密钥、Docker 或生产部署。

## 目标 / 非目标

**目标：**

- 定义一种适用于模拟和本地暂存提供商的部署提供者抽象。
- 添加从目标注册表获取的、感知目标的 deploy/build 配置。
- 添加一个本地暂存提供商，用于构建前端目标并在本地提供构建输出。
- 持久化暂存部署状态、URL、目标元数据、源 diff/review/ 预览引用和日志。
- 基于审查、预览和目标策略安全性对暂存部署进行门控。
- 保留 P6/P7/P8/P9/P10 基线和模拟部署行为。
- 对内置演示前端或外部 Vite 示例目标进行真实的暂存部署演练。

**非目标：**

- 生产环境部署。
- 云提供商令牌管理。
- Vercel、Netlify、Render 或其他完整云提供商集成。
- Docker 或 Kubernetes。
- 域名管理。
- 数据库托管。
- 自动回滚。
- 多用户部署审批。
- 密钥存储或密钥注入。
- 用预发布部署替换预览。

## 决策

### 决策：在添加更多提供商之前，先添加提供商接口

P11 应引入一个窄化的部署提供者接口：

```text
providerId
providerType
targetId
buildCommand
deployCommand
outputUrl
status
logs
```

第一个真正的提供者是 `local_staging`；现有的模拟路径仍然保留为与提供者兼容的路径，而不是被移除。

理由：提供者接口可防止模拟部署、本地预发布以及任何未来的云部署适配器变成各自独立的临时流程。

考虑的替代方案：直接在当前模拟部署服务中添加本地暂存行为。已拒绝，因为这会模糊 mock/staging 的语义，并使未来的提供商工作更加困难。

### 决策：目标注册表拥有部署配置

目标注册表应公开与部署相关的元数据，例如：

```text
buildCommand
stagingOutputDir
stagingServeCommand
stagingBaseUrl / generated URL policy
deployProviderIds
```

外部目标可能仅在其 analyzer/registration 元数据包含足够信息以构建和提供输出时才可部署。未知目标必须如实失败或需要显式配置。

理由：P7 将目标元数据作为路径、命令和策略的真实来源。部署配置应归属于目标。

考虑的替代方案：仅将会话或任务计划上的部署配置存储。
因会分散特定目标的部署规则而被拒绝。

### 决策：本地暂存提供者先构建再提供静态输出

对于类似 Vite 的前端目标，本地预发布应：

1. 验证部署门禁；
2. 在目标根目录或会话工作树范围内执行配置的构建命令；
3. 验证配置的输出目录是否存在；
4. 在预留的本地端口上提供该输出目录的服务；
5. 对生成的预发布 URL 执行健康检查；
6. 记录状态、日志、URL、提供商、目标及源制品链接。

理由：这提供了一个真实的预发布 URL，无需云账户、令牌、Docker 或生产基础设施。

考虑的替代方案：使用 Vite 开发服务器作为预发布环境。已拒绝，因为 P11 需要与预览不同的部署制品。

### 决策：部署状态由制品和事件共同支撑

P11 应同时持久化：

- 用于摘要和 UI 卡片的 deployment/artifact 记录；
- 用于状态变更和日志的 TaskRunEvent 或等效事件记录。

状态：

```text
queued
building
deploying
ready
failed
cancelled
```

理由：制品记录使暂存部署在当前制品流水线中可见；事件保留了追踪和 SSE 兼容性。

考虑的替代方案：仅将日志存储在进程内存中。已拒绝，原因是
部署证据必须能在刷新和审计审查后保留。

### 决策：部署门控策略偏保守

当以下情况发生时，暂存部署不得继续：

- 源差异的最新审核失败；
- 当配置了预览前置条件时，预览为 failed/unhealthy；
- 审核或调度器元数据中存在目标策略违规；
- 根据注册表配置，目标不可部署；
- 请求的环境为生产环境。

警告仅在明确归类为非阻塞且由提供方结果记录时才允许。

理由：P10 提升了恢复安全性；P11 不应通过部署未经审查或违反策略的输出，重新引入不安全的执行。

考虑的替代方案：默认允许用户在有警告的情况下部署。推迟到未来审批模型出现后再实施。

## 风险 / 权衡

- **风险：本地静态服务被误认为生产环境。** 缓解措施：
  在所有位置将 provider/environment 标记为 `local_staging` / `staging`，并拒绝生产部署请求。
- **风险：构建命令执行不安全操作。** 缓解措施：从目标注册表读取命令，保持白名单机制，并拒绝未知命令。
- **风险：外部项目输出目录不统一。** 缓解措施：要求 analyzer/registration 置信度或显式部署配置。
- **风险：长期运行的预发布服务器导致进程泄漏。** 缓解措施：跟踪进程 ID、过期时间、stop/cancel 操作，并在后续任务中执行清理。
- **风险：失败的部署隐藏了有用的诊断信息。** 缓解措施：持久化日志和失败状态制品。
- **风险：P11 过度扩展至云部署。** 缓解措施：将云提供商、令牌、域名、Docker 和生产部署列为非目标。

## 迁移计划

1. 添加提供者抽象和模式，同时保留当前的模拟部署。
2. 以向后兼容的方式扩展目标注册表元数据，用于暂存部署配置。
3. 在显式暂存部署调用后添加本地暂存提供者。
4. 在启用 UI 功能前添加部署门控。
5. 使用内置演示前端或外部 Vite 示例目标进行演练。

回滚策略：保持模拟部署不变。如果预发布部署不可用，
AgentHub 可以继续暴露模拟部署和预览，同时将预发布部署报告为不支持或失败。
