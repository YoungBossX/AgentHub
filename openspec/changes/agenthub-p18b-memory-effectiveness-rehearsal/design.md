## 背景

P18 增加了规范记忆、托管指令出口、记忆快照、检索、提示注入防护、外部记忆建议、记忆界面以及确定性指标辅助工具。其冻结评审正确地指出，这些指标主要是单元级和演练级检查，而非生产环境有效性证明。

P18b 增加了一个聚焦的评估层。它使用真实的用户画像、已确认的偏好、项目规则以及有边界的 coding/chat 任务，来比较关闭记忆与开启记忆时的行为差异。目标是确定记忆是否能改善结果，并在未能改善时记录证据。

## 目标 / 非目标

**目标：**

- 定义可重复的记忆有效性演练场景。
- 支持对禁用记忆与启用记忆的运行进行 control/treatment 比较。
- 测量偏好召回率、跨智能体一致性、精确率、陈旧注入、提示注入拦截、任务成功增量、变更日志缺失率以及定性结果记录。
- 持久化或渲染一份可审计的演练报告。
- 保持评估范围可控，使其能在本地 CI 或本地冒烟测试中运行，无需实时 Claude/Codex/API 凭证。
- 当 keys/auth 可用时，允许使用真实提供者证据，但绝不允许将虚假或脚本化结果声称是真实提供者的成功。

**非目标：**

- 不要用嵌入、RRF 或图检索替换 P18 检索。
- 不要添加自动长期学习。
- 不要收集真实的个人数据或私人用户文档。
- 不允许记忆绕过目标注册表、计划验证器、护栏或运行时配置。
- 不要添加提供商市场、生产部署或新的编码适配器。
- 确定性测试不需要实时 LLM 或编码代理凭证。

## 决策

### 使用基于角色的真实场景

P18b 应使用合成但逼真的角色，例如一位偏好简洁中文摘要、要求代码变更时更新变更日志、并希望前端工作通过项目命令验证的中文母语独立开发者。这样能在不存储个人数据的情况下提供真实的记忆压力。

考虑的替代方案：仅运行任意生成的任务。这虽然易于自动化，但过于薄弱，无法展示记忆是否能改善用户特定行为。

### 比较对照组与实验组运行

每个场景应定义：

- baseline/control 记忆状态；
- 治疗记忆状态；
- 用户请求；
- 预期记忆 ID；
- 预期排除的记忆 ID；
- 目标角色与目标；
- 所需证据点；
- 评分预期。

第一个实现可以利用现有的内存快照和检索服务来模拟 Planner/coding 上下文组装。真实的提供商冒烟测试仍为可选，且需基于证据驱动。

考虑的替代方案：仅运行一条启用记忆的路径。这能证明该路径可行，但无法回答记忆是否带来了任何改进。

### 保持结果评分明确

P18b 应添加一个摘要模型，用于分离：

- 确定性 retrieval/context 指标；
- task/evidence 指标；
- provider/runtime 可用性；
- 评估器说明与限制。

当不存在可比较的实时任务或已接受的确定性代理时，任务成功差值应记录为 `unknown`。系统必须优先选择诚实的未知状态，而非虚假的正面结果。

### 复用 P18 原语

排练应复用 `MemoryItem`、`MemorySnapshot`、检索辅助函数、
上下文包生成、规划器输入生成以及现有的指标辅助函数。
P18b 应围绕它们添加编排和报告功能，而非构建并行的记忆栈。

### 冻结前报告

P18b 冻结评审应包括：

- 场景列表；
- control/treatment 指标；
- provider/runtime 可用性；
- 内存是改善、退化还是无法确定；
- 高风险延期项。

## 风险 / 权衡

- 真实提供商凭证可能不可用 → 确定性模拟客户端和上下文组装覆盖仍会运行；真实提供商成功状态仍为未声明。
- 指标可能表现良好但任务质量仍然较差 → 包含定性任务结果说明，并在证据不足时将任务成功差异标记为未知或不确定。
- 记忆可能过度拟合场景 → 至少定义三类场景：偏好回忆、项目规则合规性和提示注入防御。
- 评估框架可能演变为产品分析系统 → 保持 P18b 本地化、有边界且面向报告。
