## 背景

AgentHub 最终演示加固已冻结。当前系统是一个本地单用户 Agent 编码工作区 / 强演示 MVP，具有以下已验证的循环：

```text
requirement -> plan -> agent execution -> diff -> preview -> mock deploy
```

当前基线：

- 本地单用户工作区；
- SQLite 持久化；
- 多会话支持；
- IM 风格聊天界面；
- 针对 `@orchestrator`、`@frontend`、`@backend` 和 `@qa` 的提及路由；
- 简单确定性编排器规划器；
- 每个会话对应一个 Git 工作树；
- 同一会话内跨 TaskRun 复用工作树；
- `CodexAdapter`、`ClaudeCodeAdapter` 和 `ScriptedMockAdapter`；
- 真实文件变更、Git 差异制品、Vite React 预览及模拟部署；
- 通过 `ScriptedMockAdapter` 的强制失败兜底路径；
- 同一会话内的后续文本变更路径。

P5 应将此基线演进为本地单用户 IM 风格的多智能体编码工作区 v1。它应减少一次性演示的感觉，更像是一个用户与智能体对话、查看智能体联系人、遵循经理主导的工作流程、内联审查制品，并基于先前证据继续推进的工作区。

P5 绝不能一下子成为长期平台。多端、多用户、外部 IM 集成、用户创建智能体、提供商市场、真实部署、Docker 沙箱以及企业审批，这些都是未来路线图上的事项，而非 P5 的即时实现内容。

## 产品形态

P5 目标路径：

```text
user requirement
-> Manager Agent planning
-> coding agent execution
-> review agent
-> artifact cards
-> preview
-> mock deploy
-> follow-up interaction
```

产品概念：

- **Agent 联系人：** 显示内置 Agent 的头像、名称、角色、adapterType、能力标签和状态。
- **直接聊天模式：** 用户可直观地聚焦于单个 Agent 或服务角色。
- **群组工作流模式：** 用户可观察 Manager、Coding Agent、Review Agent、QA/Preview/Deploy 服务共同参与一个工作流。
- **Manager Agent：** PM/PMO-style 协调器，负责规划、分配、跟踪和总结工作流。
- **Coding Agent：** 当前代码适配器（如 Codex 和 Claude Code）执行有边界的写入任务。
- **Review Agent：** read-only/non-blocking v1 审查器，在代码差异生成后产出审查制品。
- **制品卡片：** 用于 Diff、Preview、Review 和 Mock Deploy 的内联卡片，每张卡片均可选择以进行后续交互。

## 设计约束

- 保留最终演示基线。
- 保留 `CodexAdapter`、`ClaudeCodeAdapter` 和 `ScriptedMockAdapter`。
- 不要破坏 diff、预览或模拟部署流程。
- 不要伪造真实代理的成功。
- 记录任何真实的 Claude/Codex 执行。
- 同会话写入任务必须保持串行，以避免工作树冲突。
- 审查任务在 v1 中可以是只读且非阻塞的。
- 动态规划必须有界，并且必须保留确定性兜底规划。
- P5 保持本地且单用户。

## P5-1 Agent 注册表与 IM 联系人界面

### 目标

将代理以即时通讯风格的联系人形式呈现，使 AgentHub 更像一个多代理工作区，而非隐藏的任务运行器。

### 必需的后端变更

- 暴露一个代理注册端点，或在现有 workspace/session 响应中丰富已启用的代理信息。
- 包含显示元数据：头像键、名称、角色、适配器类型、能力标签、状态和简短描述。
- 将现有种子角色映射为产品角色：
  - 编排器或管理代理；
  - Frontend/Backend 编码代理；
  - 质量保证或审查代理；
  - Preview/Deploy 服务角色（如果以系统联系人形式表示）。
- 保持当前适配器类型不变。

### 必需的前端变更

- 在工作区外壳中添加代理联系人列表。
- 显示头像、名称、角色、适配器类型、能力标签和状态。
- 添加可视化的直接聊天和群组工作流模式。
- 直接聊天模式在 P5 阶段可能仅限 visual/contextual；无需实现真正独立的一对一持久化语义。
- 群组工作流模式应使管理员、编码代理、审查代理和制品服务在视觉上呈现。

### 数据模型影响

- 优先扩展现有的 `Agent` 元数据。
- 如有需要，添加显示字段，例如 `avatar_key`、`capability_tags_json`、
  `status` 和 `description`。
- 在 P5 阶段不要添加多用户账户或外部联系人表。

### 安全风险

- 联系人界面可能暗示实际并不具备的能力。
- 若代理状态并非基于真实可用性得出，则可能误导用户。
- 暴露适配器细节可能泄露本地环境假设。

### 为何从最终演示强化阶段推迟

最终演示加固冻结了证据和文档。智能体交互界面变更会影响产品表面和后端 API 形态，因此应归入 P5，而非冻结的演示。

## P5-2 共享上下文与执行账本

### 目标

持久化一个紧凑、可检查的会话账本，记录当前目标、
活跃代理、最新 plan/task/run/diff/review/preview/deploy、已变更文件
以及审查摘要。

### 必需的后端变更

- 添加一个账本服务，在消息创建、规划、任务运行完成、差异收集、审查创建、预览创建和模拟部署创建之后进行更新。
- 提供当前会话执行账本的读取端点。
- 包含制品 ID 和已更改的文件，以便后续交互可以引用具体证据。
- 尽可能保持账本写入的确定性，并使其可从现有记录中推导得出。

### 必需的前端变更

- 显示当前目标和活跃的智能体。
- 显示最新计划、活跃任务、最新运行状态、已变更文件、审查状态、
  预览状态和部署状态。
- 允许用户检查账本，同时不隐藏现有的任务时间线。
- 利用账本状态来驱动后续的上下文提示标签。

### 数据模型影响

- 添加一条 `SessionExecutionLedger` 记录或一组规范化的账本快照。
- 存储对最新消息、计划、任务、任务运行、差异、审查、预览和部署制品的引用。
- 将变更的文件和审查摘要以结构化 JSON 形式存储。
- 在 P5 阶段避免使用完整向量记忆或隐藏的跨会话记忆。

### 安全风险

- 账本可能保留敏感文件名或摘要。
- 过时的账本字段可能导致后续任务定位到错误的制品。
- 若后续引入多用户场景，跨会话账本泄露将造成严重后果。

### 为何从最终演示强化阶段推迟

最终演示已能恢复消息、任务和制品。账本（ledger）是用于 P5 后续跟进与可追溯性的新产品抽象，而非冻结任务。

## P5-3 审查 Agent 工作流

### 目标

在生成代码差异后，运行审查代理任务并生成带有 `passed`、`warning` 或 `failed` 状态的审查制品。默认情况下，第一个版本为非阻塞模式。

### 必需的后端变更

- 新增审查任务类型和审查制品类型。
- 在编码 TaskRun 产生差异制品后触发审查任务。
- 打包差异元数据、变更文件及补丁文本，供只读审查使用。
- 根据配置的审查代理策略，使用 `ClaudeCodeAdapter`、`CodexAdapter` 或 `ScriptedMockAdapter`，同时保留当前编码适配器。
- 存储审查摘要、发现项、严重程度及状态。
- 保持 v1 非阻塞模式，除非后续策略明确启用阻塞关卡。

### 必需的前端变更

- 在工作流中显示审查 task/run。
- 使用 passed/warning/failed 状态渲染审查制品卡片。
- 显示审查摘要和发现。
- 将审查卡片链接到它们所审查的差异。
- 明确说明 v1 审查为 advisory/non-blocking.。

### 数据模型影响

- 添加 `Review` 或通用制品子类型支持。
- 为已审查的差异制品 ID、状态、摘要、发现 JSON、审查者代理 ID 和适配器元数据添加字段。
- 添加 run/event 链接，使审查成为执行追踪的一部分。

### 安全风险

- 审查代理输出并非安全边界。
- 漏报可能造成虚假信心。
- 误报可能产生不必要的警告。
- 审查提示可能暴露补丁内容；打包必须保持在会话工作树和受保护路径规则范围内。

### 为何从最终演示-强化阶段推迟

Final-demo-hardening 不添加运行时工作流步骤。审查 Agent 执行会改变任务流水线和制品模型，因此它属于 P5。

## P5-4 多智能体执行追踪 UI

### 目标

将 Show Manager、Coding Agent、Review Agent、QA/Preview/Deploy 服务以可见的执行轨迹形式呈现，使用户能够理解谁做了什么以及产生了哪些证据。

### 必需的后端变更

- 确保任务运行事件包含足够的 role/agent/artifact 元数据以支持追踪渲染。
- 为会话或任务图暴露适合追踪的数据结构。
- 包含用于差异收集、预览和模拟部署的系统服务步骤。
- 保留现有的基于 TaskRunEvent 的 SSE 行为。

### 必需的前端变更

- 添加多智能体追踪组件。
- 按顺序展示管理规划、编码执行、差异收集、审查、预览和模拟部署。
- 显示进行中、已完成、警告、失败和跳过状态。
- 将追踪节点链接到消息、任务、运行和制品卡片。
- 保持现有时间线可用于小型演示。

### 数据模型影响

- 优先从现有的 `Message`、`Task`、`TaskRun`、`TaskRunEvent` 和 `Artifact` 行中推导出 trace。
- 仅当推导不足时，才添加显式的 trace 事件元数据。
- 在 P5-3 落地后，包含审查制品。

### 安全风险

- 若未标注服务生成的步骤，追踪可能夸大代理的自主性。
- 缺失的事件可能使工作流看起来比实际更简洁。
- 暴露本地路径或内部流程细节可能泄露环境数据。

### 为何从最终演示强化阶段推迟

最终演示已证明该循环可行。P5 追踪 UI 提升了产品清晰度，但并非冻结基线所必需。

## P5-5 动态管理器规划器 v1

### 目标

为前端变更意图添加一个有界动态 Manager 规划器。它应输出结构化任务图，同时保留确定性兜底规划器。

### 必需的后端变更

- 引入 Manager 规划器服务边界。
- 支持带模式验证的结构化规划器输出。
- 将 v1 限制为前端变更意图和同会话后续交互。
- 限定任务数量、角色集合、目标文件和依赖深度。
- 在模型失败、输出无效或意图不支持时，兜底至确定性规划器。
- 持久化规划器推理依据和任务图元数据。

### 必需的前端变更

- 显示 Manager 生成的计划和任务图。
- 显示规划器决策依据及适用时的兜底使用情况。
- 允许用户在开始执行前检查任务分配。
- 保持当前固定演示路径的可见稳定性。

### 数据模型影响

- 添加计划图元数据或 `Plan`/`PlanVersion` 结构。
- 存储已验证的结构化任务图输出。
- 将任务关联回创建它们的 Manager 计划版本。
- 保留当前的 `Task` 和 `TaskRun` 行为。

### 安全风险

- 提示注入可能扩大范围或指向受保护路径。
- 模型输出可能将写入操作分配给错误的代理。
- 动态规划可能创建过多任务。
- 不支持的请求可能看似被接受，除非被明确拒绝。

### 为何从最终演示强化阶段推迟

动态规划会改变核心运行时行为。P5 只能将其作为带有确定性兜底的受限 v1 引入，而不能作为最终演示加固补丁。

## P5-6 制品消息卡片 v2

### 目标

将差异渲染、预览、审查和模拟部署作为内联消息卡片展示，并允许用户选择或引用制品以进行后续交互。

### 必需的后端变更

- 在消息或后续请求中添加制品引用支持。
- 提供适用于内联卡片的制品摘要负载。
- 跟踪 selected/referenced 制品 ID 以用于后续任务。
- 确保制品引用是会话作用域的。

### 必需的前端变更

- 在聊天流中新增内联制品卡片。
- 支持的卡片类型：
  - 差异对比；
  - 预览；
  - 审查；
  - 模拟部署。
- 允许用户选择卡片或将引用插入后续消息。
- 保留右侧制品面板作为详细检查器。

### 数据模型影响

- 添加消息到制品的引用记录或结构化消息元数据。
- 存储选定的制品 ID 以用于后续上下文。
- 复用现有的制品 ID 和制品子类型记录。

### 安全风险

- 若未限定作用域，制品引用可能跨会话。
- 后续任务可能基于过期或错误的制品执行操作。
- 内联部署卡片不得暗示生产环境部署。

### 为何从最终演示加固阶段推迟

最终演示加固文档并冻结现有制品面板。内联制品卡片和后续引用是 P5 阶段的产品演进。

## P5-7 端到端演练与冻结评审

### 目标

验证并冻结 P5 本地单用户 IM 风格多智能体编码工作区 v1。

### 必需的后端变更

- 此任务不涉及新的后端功能开发；仅进行证据收集以及在 P5 实施过程中发现的最终修复。

### 必需的前端变更

- 此任务不包含新的前端功能开发；仅进行证据收集以及在 P5 实施过程中发现的最终修复。

### 数据模型影响

- 此任务中不包含新的模型范围。

### 安全风险

- 夸大平台功能仍是主要风险。证据必须标注哪些是已验证、模拟支持、仅本地可用以及已推迟的内容。

### 为何从最终演示强化阶段推迟

P5 冻结在 P5 运行时任务之后进行。它不应替换已冻结的最终演示基线。
