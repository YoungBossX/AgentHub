## 1. P5 实现任务

- [x] 1.1 P5-1 智能体注册表与 IM 联系人界面
  - 目标：将内置智能体以 IM 联系人形式呈现，并引入 direct-chat/group-workflow 种视觉模式，无需添加真实的多用户账号。
  - 范围：
    - 暴露或丰富已启用智能体的元数据，用于联系人展示；
    - 以头像、名称、角色、adapterType、能力标签和状态展示智能体；
    - 在适用场景中展示管理智能体、编码智能体、审查智能体及系统服务联系人；
    - 添加直接聊天和群组工作流两种视觉模式；
    - 保留当前 `@orchestrator`、`@frontend`、`@backend` 和 `@qa` 的提及行为。
  - 验收标准：
    - 联系人列表根据后端数据或有文档记录的后端支持的注册表结构渲染内置智能体；
    - 直接聊天和群组工作流模式仅为 visual/local 模式，不声称支持多用户；
    - 在相关场景中，`CodexAdapter`、`ClaudeCodeAdapter` 和 `ScriptedMockAdapter` 保持可见或保留；
    - 最终演示基线仍可正常工作。
  - 验证：
    - 针对变更部分执行 frontend/backend 测试；
    - `pnpm check`；
    - `pnpm test`；
    - `git diff --check`。

- [x] 1.2 P5-2 共享上下文与执行账本。
  - 目标：持久化并展示当前会话的紧凑执行账本。
  - 范围：
    - 持久化当前目标、活跃代理、最新计划、最新任务、最新运行、最新差异、最新审查、最新预览、最新模拟部署、已变更文件以及审查摘要；
    - 暴露账本读取 API；
    - 在相关 message/task/run/artifact 事件发生后更新账本；
    - 保持账本可检查且限定在会话范围内；
    - 避免使用全量向量数据库内存。
  - 验收标准：
    - 重新加载后账本可重建或读取；
    - 账本引用真实的消息、任务、运行和制品；
    - 无跨会话上下文泄漏；
    - 兜底路径保持完整。
  - 验证：
    - 针对账本更新的后端测试；
    - 若此任务中添加了账本 UI，则进行前端测试；
    - `pnpm check`；
    - `pnpm test`；
    - `git diff --check`。

- [x] 1.3 P5-3 审查 Agent 工作流。
  - 目标：在编码差异之后添加一个非阻塞的审查 Agent 工作流。
  - 范围：
    - 生成编码差异后，创建或运行一个审查 Agent 任务；
    - 生成一个状态为 `passed`、`warning` 或 `failed` 的审查制品；
    - 包含摘要和发现；
    - 将审查制品链接到被审查的差异制品；
    - 默认保持 v1 版本为非阻塞；
    - 允许审查为只读。
  - 验收标准：
    - 审查制品已持久化且可见；
    - 审查状态明确为 advisory/non-blocking；
    - 差异、预览和模拟部署流程仍能正常工作；
    - 如果使用了真实的 Claude/Codex 审查执行，则需记录；
    - 如果使用了 scripted/mock 审查行为，则需标记。
  - 验证：
    - 后端审查工作流测试；
    - 前端审查卡片测试；
    - `pnpm check`；
    - `pnpm test`；
    - `git diff --check`。

- [x] 1.4 P5-4 多智能体执行轨迹 UI。
  - 目标：将 Manager、Coding Agent、Review Agent、QA/Preview/Deploy 服务展示为可见的执行轨迹。
  - 范围：
    - 从消息、任务、任务运行、事件和制品中推导轨迹；
    - 展示 Manager 规划、编码执行、差异收集、审查、预览和模拟部署步骤；
    - 将轨迹节点链接到制品卡片和 task/run 详情；
    - 展示活动、已完成、警告、失败和跳过状态。
  - 验收标准：
    - 用户能够跟踪从需求到模拟部署的完整工作流；
    - 系统服务步骤被标记为服务，而非自主智能体；
    - 轨迹通过持久化数据在重新加载后仍可保留；
    - 现有任务时间线保持可用。
  - 验证：
    - 前端组件测试；
    - 若添加了轨迹端点，则进行 backend/API 测试；
    - 浏览器冒烟演练；
    - `pnpm check`；
    - `pnpm test`；
    - `git diff --check`。

- [x] 1.5 P5-5 动态管理器规划器 v1
  - 目标：为前端变更意图添加有边界的动态管理器规划器，同时保留确定性兜底规划能力。
  - 范围：
    - 引入管理器规划器服务边界；
    - 对有边界的前端变更意图进行分类；
    - 输出结构化任务图；
    - 验证任务图结构、角色集合、目标文件、依赖深度及任务数量；
    - 对不支持的意图或无效模型输出，回退至确定性规划器；
    - 持久化规划器决策依据和图元数据。
  - 验收标准：
    - 有边界的动态规划器能够创建有效的前端任务图；
    - 确定性演示规划器仍能处理已知的登录页面路径；
    - 不支持的宽泛请求应安全失败或回退，不得声称具备任意编辑能力；
    - 同会话内的写入任务保持串行执行。
  - 验证：
    - 规划器单元测试；
    - 针对回退行为的后端集成测试；
    - `pnpm check`；
    - `pnpm test`；
    - `git diff --check`。

- [x] 1.6 P5-6 制品消息卡片 v2。
  - 目标：将 Diff、Preview、Review 和 Mock Deploy 渲染为内联消息卡片，并支持制品 selection/reference 以进行后续交互。
  - 范围：
    - 为 Diff、Preview、Review 和 Mock Deploy 添加内联卡片；
    - 保留右侧制品面板作为详细检查器；
    - 支持在后续消息中选择或引用制品；
    - 保持引用为会话级作用域；
    - 避免暗示 Mock Deploy 是生产部署。
  - 验收标准：
    - 制品卡片在聊天流中渲染；
    - selecting/referencing 一个制品会创建可检查的后续上下文；
    - 引用的制品会针对当前会话进行验证；
    - 当执行运行时，后续交互仍会产生真实的 diff/preview 证据。
  - 验证：
    - 前端 card/reference 测试；
    - 如果添加了后端元数据，则进行后端引用验证测试；
    - 浏览器冒烟演练；
    - `pnpm check`；
    - `pnpm test`；
    - `git diff --check`。

- [x] 1.7 P5-7 P5 E2E 预演与冻结评审。
  - 目标：验证并冻结 P5 作为本地单用户 IM 风格的多智能体编码工作区 v1。
  - 范围：
    - 预演用户需求 -> 管理者规划 -> 编码智能体执行 -> 审查制品 -> 制品卡片 -> 预览 -> 模拟部署 -> 后续交互；
    - 验证真实或兜底编码执行并提供证据；
    - 验证审查制品行为及非阻塞语义；
    - 验证 reload/recovery 行为；
    - 记录注意事项和验证结果；
    - 确保未意外实现任何 P5 非目标。
  - 验收标准：
    - P5 循环包含计划、任务、运行、差异、审查、预览、部署及后续制品的证据 ID（如适用）；
    - 模拟部署仍标记为模拟；
    - 真实 Claude/Codex 执行仅在实际运行时记录；
    - 最终演示基线保持不变；
    - P5 不被描述为完整的多用户 IM。
  - 验证：
    - 浏览器 E2E 预演笔记；
    - `pnpm check`；
    - `pnpm test`；
    - `git diff --check`；
    - `openspec validate agenthub-p5-platform-evolution --strict`。

## 2. P5 的明确非目标

- 完整的多用户即时通讯平台。
- 对接真实的飞书、微信、Slack、Matrix 或其他外部即时通讯系统。
- 桌面端或移动端应用。
- 提供商市场。
- 生产环境部署。
- Docker 沙箱。
- 创建 PR。
- 不受限制的任意代码编辑。
- 完整的向量数据库记忆。
- 分布式工作节点集群。
- 企业审批工作流。
- 实时多用户同步与冲突解决。
- 用户创建的智能体作为即时 P5 运行时功能。

## 3. P5 完成定义

- AgentHub 可理解为本地单用户 IM 风格的多智能体编码工作区 v1。
- 智能体联系人与工作流模式可见。
- 管理智能体的规划有边界，并具备确定性兜底。
- 编码执行仍使用保留的适配器与工作树隔离。
- 审查智能体生成非阻塞的审查制品。
- Diff、Preview、Review 和 Mock Deploy 卡片可见且可引用。
- 后续交互可引用先前的制品。
- 同会话内的写入任务仍保持串行。
- 最终演示基线仍通过。
- 未来的长期平台事项保持延期并明确记录。
