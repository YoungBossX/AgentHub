## 背景

P20 让工作区更易用，P23 引入了可编辑的制品工作台版本，P24 使部署证据可作为上下文使用。当前 UI 仍主要表现为单个选定制品加一条引用消息。日常的第二轮工作需要一个可见的托盘，能够容纳多条上下文，并以稳定、脱敏的格式传递给规划器。

相关约束：

- 普通聊天不得调用编码代理。
- 可执行的后续操作仍通过对话路由器/规划器、计划验证器、调度器和编码代理流转。
- 上下文可以引导任务，但不能授予权限或绕过目标注册表/护栏。
- 受保护路径、密钥、原始提供商令牌以及仅限主机的实现细节不得进入提供商可见的上下文。

## 目标 / 非目标

**目标：**

- 新增一个小型、可审计的上下文托盘模型，用于管理消息、制品、选中文本、部署和用户备注的上下文。
- 通过将现有引用消息和选中制品的行为映射到托盘模型中，保留其原有功能。
- 在上下文进入规划器可见范围之前，对其进行标准化和脱敏处理。
- 在聊天工作区中使上下文操作可见：添加、移除、重新排序、清除和发送。
- 记录足够的元数据，以便任务 trace/planner 证据能够展示哪些上下文影响了后续操作。

**非目标：**

- 无多用户共享上下文。
- 无向量记忆或知识图谱。
- 无任意文件上传导入。
- 无通过制品编辑直接写入仓库。
- 无绕过 PlanValidator、Target Registry 或 Guardrails 的行为。
- 无生产部署集成。

## 决策

### 上下文托盘是消息上下文，而非新的持久化表

P25 将托盘负载存储在 `Message.context_json` 上，复用了现有的消息上下文路径。这使得改动较小，且与 P20-P24 兼容。

考虑的替代方案：创建新的 `ContextTray` 表。这可以支持跨设备的草稿持久化，但在单用户本地工作流需要之前，会增加模式与生命周期复杂性。

### 归一化为 `contextItems`

前端仍可发送 `selectedArtifactId` 和 `quotedMessage`，但后端应将其归一化为名为 `contextItems` 的列表。每个条目包含：

- `id`
- `kind`
- `title`
- `summary`
- `artifactId`
- `artifactVersionId`
- `selectedText`
- `messageId`
- `deploymentId`
- `source`
- `metadata`

每种类型仅包含相关字段。

### 编辑由后端强制执行

UI 应避免发送不安全数据，但后端仍负责脱敏处理。在构建 Planner 可见的上下文之前，上下文规范化会过滤受保护键、受保护路径、类密钥字符串、原始日志和未知主机路径。

### 规划器查看摘要与引用

规划器可见的上下文应包含安全摘要、选中文本、制品 ID、部署 status/provider 元数据和来源引用。大型制品内容保留在 artifact/version API 之后，不会盲目注入。

### UI 保持轻量

聊天页面应在编辑器附近显示一个紧凑的托盘，而非厚重的配置面板。详细的制品编辑功能保留在右侧的制品工作台中。

## 风险 / 权衡

- [风险] 上下文负载可能过大。→ 对 v1 版本的项目数量、选中文本长度和摘要长度保持限制。
- [风险] 用户可能误以为上下文会改变权限。→ 将上下文标记为仅作参考材料；可执行变更仍需通过规划和防护措施。
- [风险] 旧版单选代码可能与托盘模型产生分歧。→ 将旧版字段视为兼容性输入，统一规范化为 `contextItems`。
- [风险] 类似机密的部署日志或选中文本可能泄露。→ 后端脱敏处理及测试覆盖受保护键、受保护路径和类似机密的字符串。
