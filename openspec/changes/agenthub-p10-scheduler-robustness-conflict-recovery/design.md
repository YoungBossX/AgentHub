## 背景

P8 引入了依赖感知调度、目标写锁、自动有界流水线推进、失败阻塞、retry/fallback 状态以及调度器 UI 追踪。P9 随后将执行范围从内置演示目标扩展至已注册的外部本地项目，并支持目标注册表元数据、外部 TaskRun、差异对比、命令证据和审查策略。

这种组合使得调度器的正确性更加重要。一个过时的 TaskRun 或过时的目标锁，在 `apps/demo` 中可能只是令人烦恼，但对于用户选择的外部项目来说，则可能带来风险。P10 增加了所需的健壮性层，之后 AgentHub 才应鼓励更广泛地使用外部项目。

当前约束：

- AgentHub 仍为本地单用户 Agent 编码工作区。
- SQLite 仍作为本地演示持久化层。
- 现有适配器保持不变：`CodexAdapter`、`ClaudeCodeAdapter`、
  `ScriptedMockAdapter`。
- P8 调度器语义和 P9 外部工作区行为必须保留。
- 外部目标 allowed/denied 路径仍强制生效。
- 恢复操作必须可审计，且不得伪造成功。

## 目标 / 非目标

**目标：**

- 通过心跳、租约和运行器身份追踪 TaskRun 活跃度。
- 检测僵死的运行中 TaskRun，并如实标记超时。
- 将目标锁绑定到 TaskRun 所有者，仅清理僵死的自有锁。
- 为内置和外部目标上的写入任务记录运行前检查点。
- 使重试具备足够的幂等性，避免盲目重新运行到脏的或冲突的工作树中。
- 强化下游故障传播，尤其是 preview/mock 部署门控。
- 检测文件重叠、脏工作树和契约漂移冲突。
- 为僵死运行、僵死锁、重试和下游流水线控制提供可审计的恢复操作。
- 演练僵死任务、僵死锁、失败依赖、重试和冲突场景。

**非目标：**

- 分布式工作节点集群。
- Docker 沙箱。
- 生产环境部署。
- 供应商市场。
- PR 创建。
- 全自动 Git 冲突合并。
- 多用户即时通讯。
- 企业级 RBAC。
- 修改超出 scheduler/recovery 元数据范围的适配器执行语义。

## 决策

### 决策：心跳与租约首先存活于 TaskRun 元数据

P10 应在 `TaskRun` 或等效的 TaskRun 自有元数据面上添加持久心跳字段：

```text
lastHeartbeatAt
leaseExpiresAt
runnerId
staleDetectedAt
staleReason
```

理由：P8 已将 TaskRun 视为调度器状态和目标锁的执行所有者。将活跃性锚定在 TaskRun 上可避免引入单独的 Worker 集群模型。

考虑的替代方案：创建单独的 worker/runner 表。已推迟，
因为 P10 不是分布式工作节点系统。

### 决策：目标锁是所有者绑定且重新计算的

目标锁应与 P8 锁行为保持兼容，但变为所有者绑定：

```text
targetId
taskRunId
sessionId
lockMode: write | read
acquiredAt
leaseExpiresAt
releasedAt
releaseReason
```

实现可以持久化显式锁记录，或从活跃的 TaskRun 及调度器元数据中推导出锁记录，只要过期清理能够识别所属的 TaskRun 并写入审计事件即可。

考虑的替代方案：仅使用全局目标锁行。因会导致所有权过期和清理过程难以追踪而推迟。

### 决策：写入任务需要执行前检查点

在创建或启动写入型 TaskRun 之前，AgentHub 应记录：

```text
targetId
targetRoot
allowedPaths
deniedPaths
baseCommit
gitStatus
dirtyFiles
plannedFiles
contractId
contractHash
createdAt
```

理由：重试和恢复需要一个已知的起始点。这对于可能已存在本地未提交工作的外部目标尤为重要。

考虑的替代方案：仅依赖 `base_ref`。已拒绝，因为它遗漏了脏文件、路径策略、合约漂移以及目标元数据。

### 决策：采用不安全的重试块而非猜测

重试应记录：

```text
previousRunId
failureSummary
retryMode
checkpointId
dirtyWorktreeDecision
```

如果目标在检查点或计划的安全路径之外存在脏文件，则重试必须被阻止，或需要明确的恢复决策。P10 不得自动重置或自动合并。

考虑的替代方案：始终从当前状态重试。暂缓执行，因为可能会在外部项目上累积部分适配器变更。

### 决策：冲突检测是保守的

P10 应检测冲突并停止，而非合并：

- 文件重叠冲突：两个可执行写入任务计划或触及重叠文件；
- 工作区脏冲突：目标包含检查点未预期的未提交文件；
- 契约漂移冲突：下游任务引用的契约 hash/version 与当前活跃契约不再匹配。

理由：保守的冲突检测可保护工作区，同时将 P10 实现范围控制在合理范围内。

考虑的替代方案：自动 Git 冲突解决。明确为非目标。

### 决策：恢复操作需明确且可审计

每个恢复操作都必须追加一条 `TaskRunEvent` 或等效的审计记录：

- 标记过期任务为失败；
- 释放过期锁；
- 从当前状态重试；
- 若安全则从检查点重试；
- 停止下游流水线；
- 在依赖和锁有效后恢复下游流水线。

理由：AgentHub 必须保持可审查性。未经审计的恢复会使外部项目变更难以信任。

## 风险 / 权衡

- **风险：错误的过期检测将活跃运行标记为失败。** 缓解措施：
  租约需要宽限期、运行器 ID 以及在清理前进行心跳新鲜度检查。
- **风险：过期锁清理释放了活跃锁。** 缓解措施：仅当所有者 TaskRun 处于 stale/timed 状态时进行清理，并记录一条指明所有者的审计事件。
- **风险：检查点记录敏感路径。** 缓解措施：检查点元数据必须遵守受保护路径的编辑规则，且不得包含机密内容。
- **风险：重试逻辑变得过于严格。** 缓解措施：允许显式的恢复决策，同时阻止不安全的自动重试。
- **风险：冲突检测阻止了合法的本地编辑。** 缓解措施：显示文件名、原因和恢复选项，而不是静默失败。
- **风险：P10 增加了过多的模式变更。** 缓解措施：从 TaskRun、Artifact 和 TaskRunEvent 元数据开始（在可行的情况下），仅在需要持久可查询性时才添加表。

## 迁移计划

1. 以向后兼容的方式添加 heartbeat/lease 和检查点元数据。
2. 确保没有心跳字段的现有 TaskRun 被视为遗留任务，除非它们处于活跃状态且根据新规则明显过期，否则不会自动标记为过期。
3. 在显式调度器调用后添加过期检测和清理 APIs/services。
4. 在更改默认 UI 行为之前添加重试和冲突检查。
5. 在声称冻结之前，通过受控的本地场景演练 P10。

回滚策略：P10 应可通过忽略新的健壮性元数据并保持 P8 调度器路径激活来移除。任何恢复操作都不应在没有用户明确决策的情况下破坏性地修改工作树。
