# AgentHub 架构抽象分析：ARCHITECTURE_MAP

分析范围：`sandbox-web-backend` 与 `java-data-flow`。本文只提炼工程架构思想，不复刻业务代码。

## 1. 整体形态判断

这两个项目本质上都是 **Maven 多模块的模块化单体后端**：

- 运行形态：最终由 `dataflow-web` 打成一个 Spring Boot Jar 启动，README 与 Dockerfile 都指向单 Jar 部署。
- 代码组织：不是独立微服务拆分，而是在一个后端应用中按 Maven module 与 Java package 划分边界。
- 前后端关系：后端可独立运行；README 提到前端 `dist` 可放入 `dataflow-web/src/main/resources/ui`，也可以前端单独启动并代理 `/api`，所以是“后端单体 + 可嵌入或分离的前端”。
- 仓库形态：每个仓库自己是一个后端 monorepo，包含多个 Maven 子模块、部署脚本、API 文档、SQL 初始化和示例脚本。

两者差异：

| 仓库 | 主要特征 |
|---|---|
| `java-data-flow` | 更完整的通用 DataFlow 分支，包含数据共享、连接器、联邦学习/SAIL、信用清算、风控、审计等模块。 |
| `sandbox-web-backend` | 偏沙箱/云南电网场景，保留 `sandboxenv`、`federatedcomputing`、`miniapp`、`sso` 等能力，突出沙箱任务、敏感数据、风险与黑名单。 |

## 2. 顶层目录职责

| 目录 | 类型 | 职责 | AgentHub 可抽象 |
|---|---|---|---|
| `dataflow-web` | 应用入口 / Web 壳 | Spring Boot 启动类、组件扫描、拦截器、参数注入、日志切面、全局 Web 配置、资源配置。 | AgentHub 的 API Gateway / App Shell：认证、租户上下文、统一响应、审计入口。 |
| `dataflow-modules` | 核心业务 API | Controller + Service 编排。按业务域拆包：`datasharing`、`connector`、`federatedcomputing`、`sandboxenv`、`system`、`audit`、`supermanage` 等。 | AgentHub 的业务域模块：Agent 管理、任务编排、会话、工具、审批、评估。 |
| `dataflow-core` | 基础能力层 | 子模块包括 `dataflow-aop`、`dataflow-jobs`、`dataflow-resource-manager`、`dataflow-system`，承载异常处理、调度、资源管理、权限/账号/连接器基础服务。 | AgentHub 的平台内核：任务调度、事件总线、资源/凭证/权限管理。 |
| `dataflow-common` | 公共组件 / 外部 SDK 包装 | 配置管理、动态数据源、SDK 初始化、分布式锁、线程池、上下文、工具类、事件定义。 | AgentHub 的 shared infrastructure：配置中心、外部 LLM/工具网关、租户上下文、锁与工具类。 |
| `dataflow-domain` | 持久化领域对象 | JPA `@Entity`，按业务域分包：资源、系统、权限、任务、审计、连接器、联邦学习等。 | AgentHub 的领域模型：Agent、Run、Step、ToolCall、Artifact、Workspace、Policy。 |
| `dataflow-dao` | Repository 层 | Spring Data JPA Repository、自定义查询、视图对象。 | AgentHub 的 persistence ports：Repository 接口与查询视图。 |
| `dataflow-entity` | DTO / VO / Req / Enum / ErrorCode | 请求对象、返回对象、DTO、MapStruct mapper、枚举、错误码、统一响应。 | AgentHub 的 API contract 包：请求/响应、错误码、状态枚举、领域事件 DTO。 |
| `dataflow-sso` | SSO 适配模块 | Sa-Token SSO server/client。`sandbox-web-backend` 正式引入，`java-data-flow` 中存在 target 或注释依赖。 | AgentHub 的身份供应商适配层。 |
| `dataflow-miniapp` | 可选业务扩展 | 风险管理、黑名单等小程序扩展，仅 `sandbox-web-backend` 完整存在。 | AgentHub 的插件化业务包或实验模块。 |
| `scripts` / `deploy-init-bundle` | 部署初始化 | 启停脚本、环境检查、机构初始化、数据库初始化 SQL。 | AgentHub 的 bootstrap / tenant provisioning。 |
| `docs` / `cicd` | 文档与工程辅助 | API 文档生成、项目图、部署说明。 | AgentHub 的工程自动化文档与 API contract 输出。 |
| `nginx` / `Dockerfile` | 部署适配 | Nginx 配置、Jar 容器镜像。 | AgentHub 的交付封装。 |

## 3. 核心业务与基础设施边界

### 核心业务

`java-data-flow` 的核心业务集中在：

- `modules/datasharing`：命名空间、数据库、数据源、文件、RESTful、发布/订阅/交易。
- `modules/connector`：连接器数据接入、远程数据同步、免密签名调用。
- `modules/federatedcomputing`：多方安全计算任务、发起方/参与方、模板、消息同步。
- `modules/federatedlearning` 与 `modules/sail`：联邦学习/SAIL 引擎任务、模型、预测、特征工程。
- `modules/creditliquidation`：信用清算与账本。
- `modules/riskmanagement` / `blacklist`：风控和黑名单。
- `modules/system` / `supermanage`：账号、机构、权限、后台配置。

`sandbox-web-backend` 的核心业务集中在：

- `modules/sandboxenv`：沙箱任务创建、参与方协作、算法/模型、执行、结果审核、转生产。
- `modules/federatedcomputing`：联邦计算相关任务。
- `modules/datasharing`：资源管理基础能力。
- `modules/sensitive`：敏感数据识别/查询。
- `dataflow-miniapp`：风险、黑名单。

### 基础设施

- `dataflow-core/dataflow-aop`：统一异常处理、响应包装、注解标记。
- `dataflow-core/dataflow-jobs`：Quartz 任务、JobExecutor、Disruptor 事件处理器。
- `dataflow-core/dataflow-system`：账号/机构/权限/消息/连接器等系统能力。
- `dataflow-core/dataflow-resource-manager`：资源管理服务，如 Namespace、Datasource、File、Model、Database、DataImport。
- `dataflow-common`：配置、SDK、动态数据源、线程池、事件、锁、工具。
- `dataflow-dao`：Repository 抽象。

### 外部系统适配

- Bxm/Datasys SDK：`BxmSdk`、`BxmSdkOrgMgr`，处理底层数据系统/链节点调用。
- Chainroll SDK：链上数据同步、交易、信用等。
- Connector：`ConnectorRequestServiceImpl`、`ConnectorSyncServiceImpl`，把外部连接器 API 适配为平台资源。
- FL/SAIL：`modules/federatedlearning`、`modules/sail` 与相关 SDK/API。
- 数据库适配：关系型数据库、Mongo、国产数据库、动态数据源。
- SSO/签名：Sa-Token、JWT、HMAC 签名、SM2/SM4 加解密。

## 4. 模块依赖方向

典型依赖方向如下：

```text
HTTP Request
  -> dataflow-web
  -> dataflow-modules/controller
  -> dataflow-modules/service 或 dataflow-core/service
  -> dataflow-dao/repository
  -> dataflow-domain/entity
  -> dataflow-common/component/sdk/config
  -> 外部系统或数据库
```

在 `java-data-flow/dataflow-web/pom.xml` 中，`dataflow-web` 依赖 `dataflow-modules`；`dataflow-modules` 再依赖 `core`、`dao`、`domain`、`entity`、`common`。这种方向说明它更像“单应用内的分层模块”，不是微服务间 RPC。

## 5. 工程分层识别

| 分层名 | 是否存在 | 例子 |
|---|---:|---|
| Controller | 是 | `DatasourceController`、`SBEInitiatorController`、`ConnectorController` |
| Service | 是 | `DatasourceServiceImpl`、`SBEInitiatorServiceImpl`、`ConnectorServiceImpl` |
| Repository | 是 | `DatasourceRepository`、`TeeTaskRepository`、`ExternalPublishRepository` |
| Domain Entity | 是 | `DFDatasource`、`DFTeeTask`、`DFSystemTask` |
| DTO/VO/Req | 是 | `DsCreateDTO`、`ReqCreateTask`、`DatasourceInfoVO` |
| Mapper | 是 | MapStruct：`DataSharingMapper`、`SandBoxEnvMapper`、`CommonMapper` |
| Adapter/Gateway | 是 | `ConnectorRequestServiceImpl`、`ExternalServiceImpl`、`BxmSdk`、`ChainrollSdk` |
| Task/Worker | 是 | `JobExecutor`、`OrgEventSyncJob`、`ConnectorDataSyncJob`、`SBEDatasysTaskJob` |
| Event | 是 | `BizEventData`、Disruptor handlers、`SchemaInitializedEvent` |
| Plugin-like extension | 部分存在 | `dataflow-miniapp`、`dataflow-sso`、`dataflow-modules` 下按能力接入 |
| Config Center | 内置配置管理 | `ConfigManager` 从 Spring Bean 聚合配置，支持 env override |
| Permission Layer | 是 | `PermissionManager`、`PermConfigManager`、Casbin 模型与策略 |

## 6. 对 AgentHub 的架构启发

AgentHub 不建议复刻这些业务包名，但可以继承这套“模块化单体 + 清晰基础设施层”的工程思想：

- 先做模块化单体，不急于微服务化。Agent 协作平台早期变化快，进程内调用更容易保持一致性。
- 把业务域和平台内核分开。Agent、Task、Tool、Workspace 是业务域；调度、事件、权限、配置、外部模型网关是平台内核。
- 用 Adapter 隔离外部系统。LLM Provider、Browser、Code Runner、Vector DB、GitHub、Slack 都应走统一 gateway。
- 把 `Run` / `Task` / `Step` 作为一等持久化对象，像这里的 `DFSystemTask` 和业务任务一样可查询、可重放、可审计。
- 请求上下文应自动注入。类似 `DFAccount` 注入，AgentHub 可注入 `TenantContext`、`UserContext`、`WorkspaceContext`、`TraceContext`。
