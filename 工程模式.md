# AgentHub 架构抽象分析：ENGINEERING_PATTERNS

本文从两个项目中提炼可复用工程设计模式。重点是“思路”，不是业务逻辑。

## 1. 模块化单体

项目采用 Maven 多模块组织，但最终作为一个 Spring Boot 应用部署。

值得借鉴的点：

- 业务域用 `dataflow-modules` 拆包，平台能力用 `dataflow-core` 拆模块。
- `dataflow-web` 只做启动、拦截、配置、切面，不堆业务逻辑。
- `dataflow-entity` 独立承载 API contract、DTO、VO、错误码、枚举，降低 controller/service 之间的粘连。

AgentHub 建议：

- 初期采用模块化单体：`agenthub-web`、`agenthub-modules`、`agenthub-core`、`agenthub-domain`、`agenthub-dao`、`agenthub-contract`、`agenthub-common`。
- 等业务边界稳定后，再把 worker/runtime/provider gateway 拆成独立服务。

## 2. Adapter / Gateway 隔离外部系统

典型例子：

- `ConnectorRequestServiceImpl` 封装连接器远程 API、签名、URL 规则、SaaS 路径差异。
- `ExternalServiceImpl` 把外部接口/数据源转换成内部资源。
- `BxmSdk`、`ChainrollSdk` 把底层数据系统和链上能力封进公共组件。

工程思想：

```text
业务服务不直接拼 HTTP / SDK 参数
  -> Gateway 负责协议细节
  -> Adapter 负责外部 DTO 到内部领域对象
  -> Service 只表达业务动作
```

AgentHub 落地：

- `ModelProviderGateway`：OpenAI、Anthropic、本地模型等。
- `ToolGateway`：Browser、Shell、GitHub、Slack、数据库等。
- `ArtifactGateway`：文件、对象存储、向量库。
- `CredentialAdapter`：不同供应商密钥、OAuth、租户级凭证隔离。

## 3. 上下文注入

项目通过 `AuthenticationInterceptor` 解析 token，构造 `DFAccount` 并写入 request attribute；`AccountInjectionHandler` 自动把 `DFAccount` 注入 controller 参数。

优点：

- controller 不重复写 token 解析。
- 所有业务 service 都拿到同一份账号/租户上下文。
- 结合 `DynamicDataSource.switchDataSource(account.getPoolName())` 实现租户级路由。

AgentHub 落地：

- 定义 `RequestContext`：`tenantId`、`workspaceId`、`userId`、`roles`、`traceId`、`budgetPolicy`。
- controller/service 不解析 token，只接受上下文对象。
- 所有 Run、Step、ToolCall、Artifact 写入都自动带上下文字段。

风险：

- 隐式上下文容易被滥用，测试时要提供 context fixture。
- 异步任务不能依赖 HTTP request，需要显式序列化上下文快照。

## 4. 权限隔离

项目在资源级别使用：

- `PermissionManager`
- `PermConfigManager`
- `CtlPermissionCommon`
- Casbin policy/model
- 资源路径：namespace、resource id、目录路径、操作 path。

常见动作：

- 创建资源后给创建者/管理员 owner role。
- 进入修改/删除/发布前检查权限。
- 删除资源时清理授权关系。

AgentHub 落地：

- 资源维度：workspace、agent、run、artifact、tool credential、memory。
- 动作维度：read、execute、modify、approve、share、delete、manage_members。
- 创建 Agent/Run/Artifact 后自动 grant owner。
- Agent 运行工具前执行 policy check，避免“Agent 越权调用工具”。

## 5. 统一响应与错误处理

项目通过：

- `ControllerResponseHandler` 自动把 controller 返回值包装为 `Response.success(data)`。
- `GlobalExceptionHandler` 捕获业务异常、参数异常、SDK 异常、未知异常。
- `errorcode/*Enum` 统一错误码与用户提示。
- `Assert` 风格错误码断言，例如 `assertNotNull`、`assertIsTrue`。

AgentHub 落地：

- API envelope 统一：`code`、`message`、`data`、`traceId`。
- 错误分类：validation、auth、quota、provider、tool、runtime、policy、internal。
- Run/Step 失败时同样使用结构化 error code，方便 UI 展示和重试策略。

风险：

- 自动包装可能影响文件下载、流式响应、SSE，需要白名单或特殊返回类型。

## 6. 异步任务调度

项目的任务系统有两层：

- Quartz：持久化定时任务，`DFSystemTask` 保存任务配置，`DFSystemTaskLog` 保存执行日志。
- `JobExecutor`：统一执行模板，负责状态、日志、异常、afterExecute。

启动恢复：

- `SchemaInitializedEvent` 发布后，`AppReadyEventListener` 获取分布式锁，恢复机构同步、缓存清理、审计同步、沙箱状态上报等后台任务。

AgentHub 落地：

- `RunScheduler`：恢复未完成 Run，调度超时检查、重试、清理、周期性评估。
- `RunExecutor`：统一执行模板，写 `RunLog`、`StepLog`、`ToolCallLog`。
- `SystemTask`：用于后台巡检、索引、清理、模型配额同步。

风险：

- 单纯 Quartz 适合定时任务，不适合高吞吐实时队列。AgentHub 的 run/step worker 更适合用队列或 durable execution 引擎。

## 7. 事件驱动

项目用两类事件：

- Spring 应用事件：`SchemaInitializedEvent` 表示主库初始化完成，触发配置和任务恢复。
- Disruptor 业务事件：`BizEventData` 经 RingBuffer 投递到 `OrgAndUserEventHandler`、`ResourceEventHandler`、`OrderAndAppKeyEventHandler`、`OpLogEventHandler`、`LiquidationEventHandler`、`FlEventHandler`、`SBEInitiatorServiceImpl` 等消费者。

AgentHub 落地：

- 领域事件：`RUN_CREATED`、`STEP_STARTED`、`TOOL_CALL_FINISHED`、`ARTIFACT_CREATED`、`APPROVAL_REQUESTED`。
- 事件消费者：审计、通知、指标、记忆索引、artifact 索引、回放/评估。
- 事件先落库再异步消费，保证可追踪和可恢复。

风险：

- 进程内 Disruptor 性能高，但跨进程/崩溃恢复弱。AgentHub 如果需要可靠多 worker，建议用持久队列或 outbox pattern。

## 8. 状态建模

项目大量用枚举表达业务状态：

- `SBETaskStatusEnum`
- `SBETaskPartnerStageEnum`
- `SBETaskPartnerStatusEnum`
- `SBETaskDataStatusEnum`
- `OrderStatusEnum`
- `PublishStatusEnum`

优点是简单直观；缺点是状态转移散落在 service 代码中，不易审计非法转移。

AgentHub 落地：

- Run 状态：`CREATED`、`QUEUED`、`RUNNING`、`WAITING_APPROVAL`、`PAUSED`、`SUCCEEDED`、`FAILED`、`CANCELED`、`EXPIRED`。
- Step 状态：`PENDING`、`RUNNING`、`RETRYING`、`BLOCKED`、`SUCCEEDED`、`FAILED`。
- 建议引入显式 `StateTransitionService`，所有状态变化写 `transition_log`。

## 9. 配置中心与启动初始化

项目有 `ConfigManager` 聚合 Spring 配置 Bean，并在初始化时：

- 初始化 SDK。
- 读取 root account。
- 初始化 root 账号。
- 初始化动态数据源。
- 初始化机构节点与消息处理。

配置可通过 `application.yaml` 和环境变量覆盖。

AgentHub 落地：

- `ConfigManager` 不建议做成全静态大对象，可以按领域拆：`ProviderConfig`、`RuntimeConfig`、`SecurityConfig`、`SchedulerConfig`。
- 启动顺序明确：数据库 -> tenant config -> provider registry -> scheduler -> worker recovery。

## 10. 缓存代理

项目在 `DataImportCacheManager` 中根据配置选择内存实现或数据库实现：

```text
DataImportCache interface
  -> DataImportCacheMemoryImpl
  -> DataImportCacheDatabaseImpl
  -> DataImportCacheManager 按配置代理
```

AgentHub 落地：

- `RunCache`、`ToolResultCache`、`MemoryIndexCache` 可采用 interface + impl + manager。
- 开发环境用 memory，生产环境用 Redis/DB。

## 11. 日志追踪与审计

项目做法：

- `WebLogAspect` 记录 URL、参数、调用用户、耗时、异常。
- audit 模块和 Disruptor handler 异步写操作日志。
- Quartz 写 `DFSystemTaskLog`。

AgentHub 落地：

- 每个 HTTP 请求、Run、Step、ToolCall 都带 `traceId`。
- Run 级别保存 timeline。
- 工具调用参数和结果要分级脱敏。
- 审计日志与用户可见运行日志分开存储。

## 12. 测试策略

项目有单元/集成测试目录，例如：

- `dataflow-web/src/test`
- `dataflow-dao/src/test`
- `dataflow-core/dataflow-resource-manager/src/test`
- `dataflow-core/dataflow-jobs/src/test`

但 `dataflow-web/pom.xml` 中 `maven-surefire-plugin` 配置了 `skipTests=true`，说明 CI/打包时可能默认跳过测试。

AgentHub 建议：

- Adapter 测试：mock provider/tool 响应。
- State transition 测试：非法转移必须失败。
- Run executor 测试：成功、失败、重试、取消、超时。
- Policy 测试：Agent 工具调用前后的权限边界。
- Contract 测试：API envelope 与错误码。

## 13. 部署策略

项目交付方式：

- Maven 打包得到 Jar 和 `target/config`。
- Dockerfile 基于 OpenJDK8 JRE，解压 tar 包后用 `start.sh` 启动 Jar。
- 脚本负责 start/stop/status/restart、环境检查、机构初始化。
- Nginx 配置用于前端/反向代理。

AgentHub 建议：

- 初期一个 API 服务 + 一个 Worker 进程 + 一个数据库。
- worker 独立部署，避免长任务阻塞 API。
- config、migration、tenant bootstrap 独立脚本化。
- 支持本地开发的一键启动与生产环境的容器镜像。

## 14. 最值得复用的工程思维

| 模式 | 复用价值 |
|---|---|
| 模块化单体 | 保持早期交付速度，同时维持边界清晰。 |
| Web 壳统一上下文 | 让业务代码专注业务，减少认证/租户/响应样板。 |
| Adapter/Gateway | 外部系统多变时保护核心模型。 |
| 任务表 + 调度器 + 执行日志 | Agent run 必须可恢复、可审计、可追踪。 |
| 事件驱动 | 审计、通知、指标、索引与主流程解耦。 |
| 权限资源模型 | Agent 工具权限必须可解释、可授权、可回收。 |
| 错误码断言 | 业务失败可预期、前端可展示、日志可聚合。 |
| 配置化实现选择 | memory/db/redis、mock/real provider 能平滑切换。 |
