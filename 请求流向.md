# AgentHub 架构抽象分析：REQUEST_FLOW

本文选择 3 条典型链路：数据源创建、连接器同步、沙箱任务创建/调度。它们分别代表同步 HTTP 请求、外部系统适配、异步任务/事件。

## 1. 公共请求前置链路

大多数 `/api/**` 请求先经过同一套 Web 壳：

```text
HTTP /api/**
  -> AuthenticationInterceptor
  -> DynamicDataSource / JWT / 签名 / token 校验
  -> request.setAttribute("account", DFAccount)
  -> AccountInjectionHandler
  -> Controller method 参数自动注入 DFAccount
  -> WebLogAspect 记录 URL、参数、用户、耗时
  -> ControllerResponseHandler 包装 Response.success(data)
  -> GlobalExceptionHandler 包装 Response.error(code, message)
```

关键文件：

- `dataflow-web/.../interceptor/AuthenticationInterceptor.java`
- `dataflow-web/.../interceptor/AccountInjectionHandler.java`
- `dataflow-web/.../configuration/GlobalConfigurer.java`
- `dataflow-web/.../aspect/WebLogAspect.java`
- `dataflow-core/dataflow-aop/.../ControllerResponseHandler.java`
- `dataflow-core/dataflow-aop/.../GlobalExceptionHandler.java`

这个设计的价值是：业务 controller 不需要显式解析 token、租户、响应格式和异常格式，所有横切关注点都被平台层接管。

## 2. 链路一：创建数据源

代表文件：

- `java-data-flow/dataflow-modules/.../datasharing/controller/DatasourceController.java`
- `java-data-flow/dataflow-core/dataflow-resource-manager/.../DatasourceServiceImpl.java`
- `java-data-flow/dataflow-domain/.../resource/DFDatasource.java`
- `java-data-flow/dataflow-dao/.../resource/DatasourceRepository.java`

### 入口

```text
POST /api/v2/{namespace}/data-source/create
```

入口方法：`DatasourceController.create(DsCreateDTO, namespace, DFAccount)`

### 处理过程

```text
DatasourceController.create
  -> DynamicDataSource.switchDataSource(account.poolName)
  -> BxmSdkOrgMgr.setReqContext(account.orgId alias)
  -> @Valid 校验 DsCreateDTO
  -> ctlPermissionCommon.verifyPermission(..., "/data-source/create")
  -> datasourceService.create(...)
      -> datasourceRepo.queryByName(...) 检查名称重复
      -> databaseRepo.findByNsIdAndDbHash(...) 查询数据库连接
      -> DBUtil / BxmSdk 查询表字段
      -> 字段类型转换与 DsCreateDTO 构造 DSDesc / DSArgs
      -> BxmSdk.getDataSourceService().create(...) 在底层 Datasys 创建数据源
      -> BxmSdk.getDataSourceService().query(...) 查询 hash
      -> datasourceRepo.saveAndFlush(DFDatasource)
      -> dataSummaryRepo.saveAndFlush(DFDataSummary)
      -> databaseRepo.saveAndFlush(DFDatabase.generatedDataSource + 1)
      -> SyncFinished(...) 等待链上/同步表出现 sharing data
  -> permConfigManager.getAvailResTypeRoles(..., "S", "owner")
  -> permissionManager.grantRole2Object(...) 给 root 和创建者授权
  -> 返回 dsId
```

### 校验、转换、落库、返回

| 阶段 | 做法 |
|---|---|
| 入参校验 | `@Valid`、`@NotBlank`、枚举错误码断言，例如 `DatasourceErrorCodeEnum`。 |
| 租户上下文 | 从 `DFAccount.poolName` 动态切换数据源。 |
| 权限校验 | `CtlPermissionCommon.verifyPermission` 验证 namespace 下创建权限。 |
| 数据转换 | `DsCreateDTO` 转底层 `DSDesc`、`DBField`、`DSArgs`；SDK 返回 id 后用 `DFUtil.trans2DFDataId` 转平台 id。 |
| 外部调用 | `BxmSdk` 作为底层数据系统 adapter。 |
| 落库 | 保存 `DFDatasource`、`DFDataSummary`，更新 `DFDatabase` 统计字段。 |
| 返回 | controller 直接返回 `String dsId`，由 `ControllerResponseHandler` 包装为统一响应。 |

### 异步/等待点

`DatasourceServiceImpl.SyncFinished` 会提交一个线程轮询 `sharingDataRepo.existsByDataId(dataId)`，用 `CountDownLatch` 等待本地同步完成。这不是消息队列，而是“同步请求中的异步轮询等待”，风险是请求耗时和线程占用。

### AgentHub 抽象

AgentHub 中的“注册工具/数据源/知识库”可以套用同一链路：

```text
Controller
  -> Context 注入
  -> Policy 校验
  -> Adapter 创建外部资源
  -> 本地 Resource + ResourceSummary 落库
  -> 授权 owner/member
  -> 返回资源 ID
```

## 3. 链路二：连接器数据同步

代表文件：

- `java-data-flow/dataflow-core/dataflow-system/.../connector/impl/ConnectorSyncServiceImpl.java`
- `java-data-flow/dataflow-core/dataflow-system/.../connector/impl/ConnectorRequestServiceImpl.java`
- `java-data-flow/dataflow-core/dataflow-system/.../external/impl/ExternalServiceImpl.java`
- `java-data-flow/dataflow-modules/.../connector/controller/ConnectorController.java`

### 入口

有两个入口形态：

1. 用户显式请求：

```text
POST /api/v2/wz/connector/data
```

`ConnectorController.connectorData` 调用 `connectorService.dataFromConnector(...)`，拉取某个连接器数据并转换成本平台资源。

2. 后台定时同步：

```text
ConnectorDataSyncJob
  -> ConnectorSyncService.syncConnectorData(orgId, connectorAddress)
```

该链路用于周期性查询连接器的交付数据并发布到平台。

### 处理过程：后台同步

```text
ConnectorSyncServiceImpl.syncConnectorData
  -> 构造 ReqConnectorQuery(platformId, connectorId, platformType)
  -> connectorRequest.queryDeliveryList(...) 查询隐私计算交付数据
  -> connectorRequest.queryDeliveryList(...) 查询联邦学习交付数据
  -> 合并列表
  -> 对每条 DeliveryDataDTO:
      -> orderIdentity + ctrlInstructionId 组成 externalDataId
      -> externalPublishRepo.existsByExternalDataId 去重
      -> accountRepo.findTopByUserIdContainsOrderById(providerId) 找提供方账号
      -> 根据 dataDTO.isDatasource 分流
          -> publishConnectorDatasource
              -> connectorRequest.detailDeliveryDatasource(...)
              -> ExternalService.publishDatasource(...)
              -> externalPublishRepo.saveAndFlush(...)
          -> publishConnectorRestful
              -> connectorRequest.detailDeliveryInterface(...)
              -> DeliveryInterfaceDetailDTO 转 Restful + RestfulParams
              -> ExternalService.publishRestful(...)
              -> externalPublishRepo.saveAndFlush(...)
```

### 外部系统适配

`ConnectorRequestServiceImpl` 是很清晰的 gateway：

- `buildConnectorUrl(ConnectorAddressDTO)` 统一拼接 SaaS / 非 SaaS 路径。
- `requestSig(...)` 统一通过 `SignatureRequest.requestConnector(...)` 发免密签名请求。
- 每个方法只负责一个远程 API：`queryDeliveryList`、`detailDeliveryInterface`、`detailDeliveryDatasource`、`syncDeliveryStatus`。
- 返回后做少量规范化，例如 URL trim、补 `http://`、解密数据源字段。

### 校验、转换、落库、返回

| 阶段 | 做法 |
|---|---|
| 去重 | `ExternalPublishRepository.existsByExternalDataId` 防止重复发布。 |
| 账号映射 | 用外部 `providerId` 找内部 `DFAccount`。 |
| 类型分流 | datasource 与 restful 走不同发布路径。 |
| 数据转换 | 连接器 DTO 转内部 `Restful` / `RestfulParams` / `DatasourceDetailBase`。 |
| 外部资源创建 | `ExternalServiceImpl` 负责创建 namespace、database、datasource/restful。 |
| 关系落库 | `DFExternalPublish` 保存 externalDataId 到内部 dataId 的幂等映射。 |
| 错误处理 | 单条数据 try/catch，日志记录后继续处理后续数据。 |

### 异步/重试

这条链路的异步来自定时任务 `ConnectorDataSyncJob`。失败处理是“本轮跳过，下轮再扫”，不是显式重试队列。幂等表 `DFExternalPublish` 是可重复扫描的关键。

### AgentHub 抽象

AgentHub 可以把外部工具、知识库、企业系统都当作 connector：

```text
ConnectorJob
  -> ConnectorGateway.queryRemoteAssets
  -> AssetImportService.importIfAbsent(externalAssetId)
  -> Adapter-specific DTO -> AgentHub Resource
  -> external_resource_mapping 落库
```

重点不是连接器本身，而是“远程资产同步 + 幂等映射 + Adapter 隔离”。

## 4. 链路三：沙箱任务创建与后台状态同步

代表文件：

- `sandbox-web-backend/dataflow-modules/.../sandboxenv/controller/SBEInitiatorController.java`
- `sandbox-web-backend/dataflow-modules/.../sandboxenv/service/impl/SBEInitiatorServiceImpl.java`
- `sandbox-web-backend/dataflow-core/dataflow-jobs/.../SystemTaskService.java`
- `sandbox-web-backend/dataflow-core/dataflow-jobs/.../AppReadyEventListener.java`
- `sandbox-web-backend/dataflow-core/dataflow-jobs/.../JobExecutor.java`
- `sandbox-web-backend/dataflow-core/dataflow-jobs/.../QuartzJobService.java`

### 入口 A：用户创建沙箱任务

```text
POST /api/v2/run-env/initiator/task/create
```

```text
SBEInitiatorController.createTask
  -> DynamicDataSource.switchDataSource(caller.poolName)
  -> SandBoxEnvMapper.toTaskDTO(reqCreateTask)
  -> sbeInitiatorService.createTask(caller, taskDTO)
      -> getCurrentSourceUserIdentity(caller)
      -> existsByTaskName 检查重名
      -> DFUtil.generateSBETaskID()
      -> 构造 DFTeeTask，状态 WAIT_PERFECT
      -> taskRepository.saveAndFlush(dfTeeTask)
      -> getOrgUserInfo 填充发起方展示信息
      -> sbeTaskRecordService.addRecord(...) 写操作记录
      -> sandboxStatusReportService.taskReport(...) 上报任务信息
  -> permissionManager.grantRole2Object(...) 授权 owner/member
  -> 返回 CreateTaskVO
```

### 入口 B：应用启动后恢复后台任务

启动链路不是 HTTP，而是 Spring 事件：

```text
PrimaryDatasourceInitializer
  -> publishEvent(new SchemaInitializedEvent)
  -> ConfigManager.onApplicationEvent 初始化 SDK / root / 动态数据源
  -> AppReadyEventListener.onApplicationEvent
      -> 分布式锁 MyDistributedLock 或 RedissonLock
      -> setupOrgBackendDataSyncTasks()
          -> 为机构恢复 org-event-sync-job
          -> createUserRecordSyncTask
          -> createDBCacheCleanTask
          -> createTaskStateQuerySyncTask
          -> createSandboxStatusReportTask
          -> createTaskInfoReportTask
          -> 恢复未过期预处理任务
```

### Quartz 执行链路

```text
SystemTaskService.createAndScheduleJob
  -> systemTaskRepo.saveAndFlush(DFSystemTask)
  -> QuartzJobService.task2QuartzJob
  -> QuartzJobService.buildQuartzJobWithParams
      -> 创建或更新 CronTrigger
      -> JobDetail 放入 QuartzJob
      -> BaseJob / NoConcurrentBaseJob 执行
  -> JobExecutor.execute
      -> execute(dataMap)
      -> afterExecute(dataMap)
      -> 写 DFSystemTaskLog
      -> 更新 DFSystemTask 状态
```

### 状态与事件

沙箱任务用枚举建模状态，而不是显式状态机类：

- `SBETaskStatusEnum`：`WAIT_PERFECT`、`WAIT_DEPLOY`、`PREPARE`、`EXECUTING`、`FINISH`、`TERMINATED`、`RESULT_WAIT_REVIEW` 等。
- `SBETaskPartnerStageEnum`：数据配置、算法审核、结果审核。
- `SBETaskPartnerStatusEnum` / `SBETaskDataStatusEnum`：参与方数据/算法状态。

`SBEInitiatorServiceImpl` 同时实现了 `EventHandler<BizEventData>`，可作为 Disruptor 消费者处理链上/业务事件。

### AgentHub 抽象

AgentHub 的多 Agent run 可以借鉴这条链路：

```text
POST /runs
  -> RunController
  -> RunService.createRun
      -> Run 状态 CREATED/WAITING
      -> RunParticipant / RunInput / RunPolicy 落库
      -> OperationLog 落库
      -> EventBus.publish(RUN_CREATED)
  -> PermissionService.grant owner/member

RunScheduler
  -> 恢复未完成 run
  -> 按 cron / queue / event 调度 worker
  -> RunExecutor.execute(step)
  -> 写 RunLog / StepLog / Artifact
  -> 状态推进
```

## 5. 关键链路的通用模式

| 关注点 | DataFlow 做法 | AgentHub 建议 |
|---|---|---|
| 入口统一 | `/api/**` 拦截器 + 参数注入 | API middleware + context resolver |
| 租户隔离 | `poolName` 动态数据源 | tenant/workspace 级数据隔离，可先逻辑隔离，后物理拆分 |
| 权限 | 资源路径 + Casbin role | workspace/resource/action policy |
| 业务落库 | domain entity + repository | run/task/tool/artifact/event repository |
| 外部系统 | SDK/Gateway 包装 | provider/tool adapter 层 |
| 异步 | Quartz + Disruptor + 线程池 | queue + scheduler + event bus |
| 幂等 | externalDataId 映射表 | external_resource_mapping / idempotency_key |
| 状态 | 枚举状态字段 | 显式 Run/Step 状态机，记录 transition |
| 响应 | ResponseBodyAdvice 自动包装 | API envelope + error code |
